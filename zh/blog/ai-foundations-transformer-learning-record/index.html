从一个真正的困惑开始

这次记录来自人工智能基础课程的 Transformer 机器翻译作业。因为错过了 Transformer 那节课，所以先从课件和下发代码重新补齐架构理解，再回到作业里的 Tensor shape、Encoder/Decoder、EOS、embedding 最近邻、temperature 和超参数实验。

我现在发现，真正卡住我的并不是“Transformer 有哪些模块”，而是一个更底层的问题：如果 RNN 靠一个沿时间更新的 memory vector 来保存上下文，那么 Transformer 不这样顺序递推时，它的“状态”到底在哪里？更具体地说，一次完整预测到底怎样从 memory、目标前缀，走到最后的 H[-1] 和 logits[-1]？

所以这篇笔记不想写成通用教程。它更像是我把几个误解慢慢拆开的过程：从 RNN memory 的直觉出发，理解 Transformer 怎样把一组 token 表示变成上下文状态；再回到作业代码，看这些理解怎样落到 tgt_input、memory、logits 和 loss 上。

我最初是用 RNN 的 memory 来想这件事

我对 RNN 的回忆是：它维护一个 memory vector，沿着序列顺序往前跑。每来一个 token，就用当前输入和上一时刻的 memory 算出新的 memory，同时也可以产生当前位置的输出。

可以写成：

h_t = f(x_t, h_{t-1})
y_t = g(h_t)

这里：

  • x_t 是第 t 个位置的输入向量。
  • h_t 是第 t 个位置更新后的 hidden state，也就是我说的 memory vector。
  • y_t 是基于当前 hidden state 得到的输出。

所以 RNN 的信息流是顺着时间走的：

x_1 -> h_1 -> h_2 -> h_3 -> ... -> h_t

这种结构符合直觉，因为它像人在从左到右读句子。但问题是，远处的信息要经过很多步传递，容易衰减；而且每一步依赖上一步，训练也不容易并行。

Transformer 的状态不再是一根线

我现在理解的 Transformer，不是沿着序列维护一个滚动 memory，而是把整个输入序列看成一个矩阵，让每个位置都能和其他位置直接建立联系。

更准确地说，它不是把上下文压进一个单独的 memory vector，而是保留一整组位置表示。每个位置都有自己的 hidden state；attention 决定当前位置应该从其他哪些位置拿信息，V 提供真正被聚合的内容。这样得到的上下文不是“全句唯一的摘要”，而是“每个位置各自的上下文化状态”。

如果当前序列有 N 个 token，每个 token 的向量维度是 D，那么输入表示可以写成：

X: [N, D]

这里：

  • N 是当前输入 token 数。
  • D 是 embedding 或 hidden 表示维度。
  • X[i] 是第 i 个 token 的向量表示。

Transformer 的参数主要针对固定的表示维度 D 来设计，而不是针对固定长度 N。所以输入句子长短变化时，N 可以变，attention 中间矩阵会跟着变大或变小，但模型里的 W_Q、W_K、W_V 这些参数不需要因为句子长度变化而重新训练。

我一开始说“每个 token 被拆成一个固定维度的 vector”，更准确地说应该是：

文本 -> tokenizer 切成 token -> token id -> embedding 层查表得到固定维度向量

Q/K/V 不只是三个方便记忆的名字

Self-Attention 的底层实现，是用三个可训练矩阵把输入表示 X 投影成三种角色：

Q = X W_Q
K = X W_K
V = X W_V

这里：

  • Q 是 query，表示当前位置想找什么信息。
  • K 是 key，表示每个位置提供给别人匹配的标签。
  • V 是 value，表示每个位置真正提供的内容。
  • W_Q、W_K、W_V 是可训练参数。

我需要注意的一点是：不是每个 token 自己有一套 Q/K/V 参数，而是所有 token 共享同一层里的 W_Q、W_K、W_V。每个 token 先查到自己的 embedding，再通过这些共享矩阵变成自己的 Q/K/V 表示。

一开始我只觉得 Q/K/V 是一种方便记忆的命名：Q 像“我要找什么”，K 像“我拿什么被别人匹配”，V 像“真正提供出去的内容”。但更深一层看，它把 attention 拆成了两件事：先用 Q 和 K 决定相关性，再用相关性去聚合 V 里的内容。如果只说“当前位置从别的位置拿信息”，就容易把“怎么匹配”和“拿到什么”混在一起。

相关性矩阵来自：

scores = Q K^T

如果：

Q: [N, d]
K: [N, d]

那么：

Q K^T: [N, N]

这个 [N, N] 矩阵不是固定参数，而是对当前输入动态算出来的。第 i 行表示第 i 个位置作为 query 时，它和所有位置的 key 的相关性。

接下来经过缩放和 softmax：

A = softmax(Q K^T / sqrt(d))

这里：

  • A 是 attention 权重矩阵，shape 是 [N, N]。
  • A[i, j] 表示第 i 个位置应该从第 j 个位置拿多少信息。

然后用这个权重矩阵去加权汇总 V：

H = A V

如果：

A: [N, N]
V: [N, d_v]

那么：

H: [N, d_v]

这里的 H 是每个位置融合上下文后的新表示。它和 V 是同类、同 shape 的量，但语义上已经不是原来的 V[i]，而是第 i 个位置吸收上下文信息后的表示。

我现在觉得这里的关键是：Transformer 并不是用循环让每个词一个个访问其他词，而是用矩阵乘法一次性算出所有位置之间的关系。

长度变了，为什么参数不用跟着变

我之前问：如果输入信息不同，N 不一样，参数要不要重新训练？现在的理解是：通常不需要。

因为可训练参数主要长这样：

W_Q: [D, d]
W_K: [D, d]
W_V: [D, d_v]
W_O: [h * d_v, D]

这些参数依赖的是表示维度 D、head 维度和头数，而不是序列长度 N。

当输入长度是 5 时：

X: [5, D]
A: [5, 5]

当输入长度是 20 时：

X: [20, D]
A: [20, 20]

A 会变化，但它是运行时的中间结果，不是训练参数。真正的限制主要是最大位置编码长度、训练时见过的长度分布，以及 attention 的平方级计算量。

Decoder 的关键误解：不是用未知词做输入

我一开始困惑的是：既然模型要预测下一个 token，那么当前位置的 token 不是未知的吗？未知 token 怎么参与 attention？

后来理清了：机器翻译里的 decoder 不是用未知 token 算 attention，而是用已经知道的目标语言前缀来预测下一个 token。

训练时，正确目标句子会错开一位：

tgt_input: <bos> A group of people
tgt_out:       A group of people <eos>

代码里就是：

tgt_input = tgt[:-1, :]
tgt_out = tgt[1:, :]

所以 decoder 第 i 个位置看到的是 tgt_input 到第 i 个位置为止的已知内容，目标是预测 tgt_out 中对应的下一个 token。

这也解释了为什么需要 mask。训练时完整目标句子已经在 batch 里了，如果不加 mask，第 i 个位置可能偷看未来答案。tgt_mask 的作用就是让第 i 个位置只能看 0..i 的目标前缀，不能看后面的 token。

用前缀矩阵预测下一个 token

我现在对 decoder 的理解可以写成：

源句 memory + 目标端已知前缀
-> decoder 反复加工
-> 得到 H: [N, D]
-> 用最后一个位置 H[-1] 预测下一个 token

假设推理时已经生成：

<bos> A group

定义：

  • N 是当前前缀长度，这里是 3。
  • D 是隐藏表示维度。
  • X 是目标语言前缀的输入表示矩阵，shape 是 [N, D]。
  • H 是 decoder 输出的上下文化表示矩阵，shape 是 [N, D]。
  • V_tgt 是目标语言词表大小。
  • logits 是每个位置对目标词表的预测分数，shape 是 [N, V_tgt]。

流程是：

X: [N, D]
H = decoder(X, memory): [N, D]
logits = generator(H): [N, V_tgt]

其中：

  • memory 是 encoder 对源语言句子的输出表示。
  • H 是 decoder 在“目标端当前前缀”和“源句 memory”共同作用下形成的新状态表示。
  • H[0] 表示看过 <bos> 后的状态。
  • H[1] 表示看过 <bos> A 后的状态。
  • H[2] 表示看过 <bos> A group 后的状态。

推理时只用最后一行：

H[-1] -> logits[-1]

这里的 -1 不是数学里的 H - 1，而是 Python/PyTorch 里的索引写法，意思是取最后一个位置。如果当前前缀长度是 N = 3，那么 H[-1] 就是 H[2]。它表示：在已经看过 <bos> A group，并且通过 encoder-decoder attention 看过源语言句子之后，最后一个位置形成的状态。

这个状态还不是下一个 token 本身。它更像是“基于当前所有已知信息，下一步应该输出什么”的状态表示。后面的 generator 再把它投影到整个目标词表上，得到 logits[-1]。

所以并不是整个 [N, D] 矩阵整体映射成一个概率表，而是每一行都映射成一个词表分数表：

H:      [N, D]
logits: [N, V_tgt]

训练时，每一行都可以并行监督：

H[0] -> 预测 A
H[1] -> 预测 group
H[2] -> 预测 of
...

推理时，只关心最后一行，用它生成下一个 token，然后把生成结果接回前缀，再重复这个过程，直到生成 <eos> 或达到最大长度。

这里之前有一段我没有完全接上：既然 H 是一个矩阵，为什么推理时通常只用最后一行？现在可以这样理解：推理时，前缀里的旧位置已经对应了我们过去生成过的 token，再重新看 H[0]、H[1] 没有意义；当前真正要回答的问题是“在整个已生成前缀之后，下一个 token 是什么”。这个问题对应的就是最后一个位置的状态，所以使用 logits[-1]。

从状态到 token：最后还是一个词表分类

最终输出不是一个自然语言解释，而是一个分类问题。

对于每个目标位置，decoder 先给出一个状态表示：

H[i]: [D]

然后 generator 线性层把它投影到目标词表大小：

logits[i] = H[i] W + b

这里：

  • H[i] 是第 i 个位置的上下文状态表示。
  • W 和 b 是 generator 的可训练参数。
  • logits[i] 是长度为 V_tgt 的分数向量。

每一维对应目标词表里的一个 token。训练时用 CrossEntropyLoss，它内部会处理 softmax/log-softmax，并让正确 token 的分数更高。

推理时最关键的是最后一行：

h = H[-1]              # [D]
logits[-1] = h W + b   # [V_tgt]

如果 D = 512，目标词表大小是 10000，那么 H[-1] 是一个长度为 512 的状态向量，logits[-1] 是一个长度为 10000 的分数向量。logits[-1][i] 表示“下一个 token 是词表第 i 个 token”的未归一化分数。之后才是 softmax、temperature、采样或取最大值。

所以 Transformer 的很多部分都是在构造一个更好的状态表示，最后再把这个状态表示映射到词表概率上。这里补上的接口对我很关键：模型不是直接从 attention 矩阵里吐出一个词，而是先形成状态，再用状态给整个词表打分。

从翻译到通用大模型：memory 的类比要换一下

上面这套理解首先是在机器翻译的 encoder-decoder 结构里成立的：

源语言句子 -> Encoder -> memory
目标语言前缀 + memory -> Decoder -> 下一个目标语言 token

这里的 memory 是源句经过 Encoder 后得到的状态矩阵。Decoder 通过 cross-attention 去读它。

但更通用的聊天式大模型通常不是把 user 输入单独换成一个 encoder memory。很多大模型是 decoder-only 结构，它更像是：

system prompt + user input + assistant 已生成前缀
-> causal self-attention
-> 最后位置 hidden state
-> logits
-> 下一个 token

也就是说，user 输入、历史对话和 assistant 已经生成的内容，一般都放在同一个自回归上下文里。后面的 assistant token 通过 causal self-attention 看到前面的 user/system/history token，而不是通过一个单独的 encoder memory 去读取。

所以我现在会把二者这样对应：

翻译里的源句 memory
≈ decoder-only 大模型里，当前位置之前可被注意到的上下文 token

这个类比有用，但不能完全等同。机器翻译里的 memory 是 Encoder 输出，Decoder 用 cross-attention 读取；decoder-only 大模型没有这条单独的 encoder memory 通道，而是把输入和输出前缀放进同一条序列里。

“目标语言 token”这个说法在通用大模型里也要换一下。翻译任务是在预测目标语言句子的下一个 token；聊天模型更基本地说是在预测当前上下文之后的下一个 continuation token。在对话语境里，这个 continuation 通常就是 assistant 回答的下一个 token。

我现在形成的整体直觉：状态表示才是主线

我现在觉得，神经网络里很多东西都可以这样看：

找到一种状态表示；
设计可训练参数；
让任务目标可以端到端反向传播；
用数据把参数训练出来。

Transformer 也是这个思路。它的特殊之处在于：状态表示不是靠 RNN 一步步传出来的，而是靠 attention 让不同位置直接交互出来的。

RNN 更像：

当前位置状态 = 当前输入 + 上一步 memory

Transformer 更像：

当前位置状态 = 当前 token 从整个可见上下文里按权重拿信息后形成的新表示

对于机器翻译，decoder 的当前位置状态还会融合源语言句子的 memory。最后这个状态被映射成目标词表里的下一个 token 概率。

这套算法一开始不太符合直觉，因为人会倾向于想象“应该先有一个整体上下文向量，再预测下一个词”。但 Transformer 的实际做法是让每个位置都产生自己的上下文状态：

H[i] = 看到当前位置为止的上下文状态
H[i] -> 预测下一个 token

这和 RNN 在功能上有相似性，只是 H[i] 的获得方式从顺序递推变成了 attention 聚合。

memory 这个名字反而容易误导

回到机器翻译作业时，memory 这个变量名很容易让我又想回 RNN。它们确实有相同点：都在承载上下文信息。但差别更重要。

RNN 的 memory 更像一条顺序递推出来的压缩链，前面的信息要经过很多步传到后面；Transformer encoder 的 memory 是一整段源语言序列的表示矩阵，shape 是 [S, B, E]。Decoder 不是读取一个被压成单向量的上下文，而是通过 cross-attention 回看这组源语言位置表示。

这里的“平权”是一个对我有帮助的直觉：源语言中每个位置都保留着自己的表示，decoder 当前状态再根据相关性动态决定要看哪些位置。它不是先把整句压成一个固定长度摘要，再把这个摘要交给 decoder。

为什么这套结构会有用

我现在还没有深入读理论论文，但可以先记录几个直观原因：

  1. 它让远距离 token 之间的交互路径变短。RNN 中远距离信息要经过很多步，Transformer 中可以在一层 attention 里直接建立联系。
  2. attention 矩阵是根据当前输入动态算出来的，所以它不是固定连接，而是根据内容决定谁应该看谁。
  3. 它的核心计算是矩阵乘法，适合 GPU/NPU 并行和规模化训练。
  4. 多层堆叠后，每个位置的表示会反复吸收上下文，逐渐形成更强的语义状态。
  5. 最终训练目标很直接：让正确 token 的概率更高。架构复杂，但损失函数可以端到端推动 embedding、attention、FFN 和 generator 一起更新。

但我也不想把这篇写成“Transformer 为什么伟大”的总结。现在更真实的状态是：主线已经能跑通，几个边角还需要继续用实验和代码去确认。比如 <eos> 被删掉后为什么模型会失去停止能力，需要回到样本构造和 tgt_out 看它原本在哪里学到“现在应该结束”；temperature 也不能简单理解成训练时跳出局部最优，它更准确地是在推理时改变从 logits 采样下一个 token 的分布。

我还想到一个后续可以单独写的方向：Transformer 之前一定有很多模型架构尝试。只看成功架构容易把它当成理所当然；如果能回头看一些失败者为什么失败，可能更能理解 Transformer 为什么成为现在这个形状。后面不只是问“Transformer 为什么成功”，也可以问过去的序列模型到底卡在哪里、哪些设计试图解决长距离依赖或表示瓶颈、它们又留下了什么问题。

回到作业文档：数据流和 shape

作业报告里已经把 nmt_de2en.py 的主要 Tensor 形状整理出来了。为了把前面的理解落到代码上，可以先采用作业文档中的记号：

  • B 表示 batch size。
  • S 表示源语言序列长度。
  • T 表示目标语言输入序列长度。
  • E 表示词嵌入或 hidden 表示维度。
  • V_tgt 表示目标语言词表大小。

主要数据流可以整理成：

Tensorshape含义
src[S, B]源语言 token id 序列
tgt[T + 1, B]目标语言 token id 序列
tgt_input[T, B]输入 Decoder 的目标语言前缀，由 tgt[:-1, :] 得到
src_emb[S, B, E]src 的词嵌入结果，并加上位置编码
tgt_emb[T, B, E]tgt_input 的词嵌入结果，并加上位置编码
src_mask[S, S]控制源语言 token 之间哪些位置可以互相注意，本实验中为全 0 矩阵
tgt_mask[T, T]Decoder 自注意力的未来遮挡 mask，防止当前位置看到后面的目标 token
src_padding_mask[B, S]标记源语言 batch 中哪些位置是 padding，避免模型关注 <pad>
tgt_padding_mask[B, T]标记目标语言输入 batch 中哪些位置是 padding
outs[T, B, E]Transformer Decoder 输出的目标语言上下文化表示
logits[T, B, V_tgt]每个目标位置在目标语言词表上的预测分数，由 generator(outs) 得到
tgt_out[T, B]训练标签，由 tgt[1:, :] 得到，也就是 tgt_input 向后错开一位后的下一个 token

这张表把我前面对 decoder 的理解具体化了：Decoder 不是拿“未知的下一个词”当输入，而是拿已经知道的目标语言前缀 tgt_input 作为输入，然后输出每个位置的预测分数 logits，再和错开一位的真实标签 tgt_out 计算 loss。

evaluate() 和 translate() 的区别

作业文档里已经写清楚了这两个函数的差别。

evaluate() 用在验证集评估中。它能看到数据集中已经给出的完整目标句子，所以使用 teacher forcing：先把 tgt 切成 tgt_input = tgt[:-1, :] 和 tgt_out = tgt[1:, :]，再一次性并行计算所有目标位置的 logits，最后用 tgt_out 计算交叉熵 loss。它的目的主要是得到验证集 loss，而不是真的逐词生成一整句译文。

translate() 是推理生成过程。它只有源语言输入，没有真实目标句子。程序会先把 Decoder 输入初始化为 <bos>，然后用当前已经生成的目标前缀预测下一个 token，再把预测结果接回前缀，继续预测，直到生成 <eos> 或达到最大长度。因此它是自回归逐词生成。

这个区别也解释了为什么训练和推理看起来不一样：训练/验证时可以并行算很多位置的监督信号；真正翻译一句话时，后面的 token 必须等前面的 token 先生成出来。

Encoder 和 Decoder 用到的信息

作业报告根据 transformer.py 把 Encoder 和 Decoder 的信息来源整理成了下表：

信息EncoderDecoder
src_emb是否，Decoder 不直接使用 src_emb，而是使用 Encoder 输出的 memory
tgt_emb否是
src_mask是否
tgt_mask否是
memory否是

这里的 memory 是 Encoder 对源语言句子的输出表示。它的 shape 是：

memory: [S, B, E]

原因是 Encoder 的输入 src_emb 是 [S, B, E]。Encoder 每一层会做源语言内部的 self-attention、残差连接、LayerNorm 和前馈网络，但不会改变序列长度、batch size 或 hidden 维度，所以最后输出的 memory 仍然是 [S, B, E]。

在 nmt_de2en.py 的 Base 配置中：

NUM_ENCODER_LAYERS = 3
NUM_DECODER_LAYERS = 3

所以作业里的 Base 模型是 3 层 Encoder 和 3 层 Decoder。

作业报告里还需要贴单层 Encoder 和 Decoder 的 forward 代码；但放在这篇文章里，直接贴整段代码反而会打断主线。我现在更想保留的是它们的结构差异。

单层 Encoder 比较像：

src -> self-attention -> residual + norm -> FFN -> residual + norm

单层 Decoder 多了一步 encoder-decoder attention。它先对目标语言前缀做 masked self-attention，再用 memory 做 cross-attention，最后进入前馈网络：

tgt -> masked self-attention -> cross-attention(memory) -> FFN -> tgt

这部分能帮助我区分两个 attention：Decoder 里的第一段 self-attention 只看目标语言前缀，第二段 multihead attention 才是把目标语言状态拿去和源语言 memory 对齐。

实验先写边界，不抢写结论

作业文档后面还有几类实验观察：

  • 删除 <eos> 后重新训练，并比较翻译是否能正常停止。
  • 用 --nearest 查询 embedding 空间里的最近邻 token。
  • 用 --translate --temperature 比较不同 temperature 下的生成差异。
  • 用 Tiny / Base / Wide 三组超参数比较训练速度、loss、显存占用和过拟合情况。

这些内容现在还不能全部写成结论。尤其是“超参数修改”部分，有一部分训练还没有完成；在表格结果补齐之前，只能先写实验设计和观察指标，不应该提前总结 Tiny、Base、Wide 的效果。
