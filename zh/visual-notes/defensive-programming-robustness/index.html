从“算对”到“扛得住”

这节课的主题是面向健壮性的设计。它不是单纯讲 C++ 语法，而是在回答一个更实际的问题：程序在理想输入下运行正确还不够，当用户输入奇怪内容、文件损坏、网络失败、内存申请失败，或者调用者传入不满足前置条件的数据时，程序应该怎样保持可控。

正确性和健壮性的区别

软件的正确性（Correctness）是指软件按照规约执行的能力。它要求软件行为严格符合规约定义，并且永不给用户错误结果。正确性更多面向内部实现：如果输入不满足规约，程序直接拒绝或结束，也可以是正确的。

02 正确性和健壮性

概念对比：正确性关注规约内做对，健壮性关注规约之外仍保持可控。

对应正文

正确性和健壮性的区别
  • 软件的正确性（Correctness）是指软件按照规约执行的能力。它要求软件行为严格符合规约定义，并且永不给用户错误结果。正确性更多面向内部实现：如果输入不满足规约，程序直接拒绝或结束，也可以是正确的。
  • 软件的健壮性（Robustness）是指在异常情况下软件仍能正常运行的能力。这里的异常情况，是指未被规约覆盖、超出规约定义的情形。健壮性要求程序尽量做出恰当反应，而不是总是退出。
  • 课程里的几个例子很直观：视频文件有坏帧时，正确性做法可以是停止播放并提示文件损坏；健壮性做法是跳过坏帧，从下一处正确帧继续播放。用户输入奇怪格式日期时，正确性做法是提示输入错误；健壮性做法是尝试识别不同日期格式，并告知解析结果。

03 软件中的问题链条

术语分层：主链条是 Mistake -> Fault(Bug) -> Error -> Failure，Defect 和 Exception 不应混入同一箭头链。

对应正文

软件中的“问题”
  • 课程区分了若干容易混用的词。这里的重点不是背英文，而是理解它们描述的是不同层次的问题。
  • Problem 泛指各种不正确情况。Mistake 指人的过失，是程序员在开发过程中的错误行为。Defect 是软件中的内在缺陷，泛指设计和实现中存在的问题，是导致 bug 的根源。Bug 通常和 Fault 表示同样含义，主要指代码实现中的故障。Error 是程序运行时的不正确内部状态。...
  • Defect 不应该被简单并入这条箭头链。程序员的 mistake 可能导致 defect，例如数组下标越界、访问空指针；但并非所有 defect 都来自程序员过失。外部输入不满足前置条件，而程序没有处理，也可能导致 failure，比如除数为 0、动态内存分配空间过大、读取的文件不存在。
一个 Fault 如何变成 Failure
  • PPT 用查找数组元素的例子说明：一个 fault 不一定每次都会导致 failure。
  • 这里的 fault 是循环条件 i > 0，它漏掉了下标 0。当 x=[] 时，循环不执行，fault 没被触达。当 x=[1,2,3], n=3 时，fault 代码被执行，但一开始就找到目标，没有引发错误状态。当 x=[1,2,3], n=5 时，程序没有遍历首个元素，内部状态是错的，但返回值仍是 -1，外部看不到 failure。...
  • 这对应 RIP 模型：failure 需要同时满足可达性、感染性和传播性。程序必须执行到 fault；执行 fault 后内部状态必须变成 error；error 还必须传播到输出，才能被外部观察为 failure。健壮性设计的价值，就是尽量在 error 传播成 failure 之前发现、处理或隔离它。

04 Fault 如何变成 Failure

RIP 模型：x=[] 未触达，n=3 触达但未感染，n=5 感染但未传播，n=1 才成为外部可见 failure。

对应正文

提高健壮性的三类方法
  • 第一类是故障拒绝（Fault Avoidance）：在设计和开发时减少错误或缺陷，例如防御性编程、代码审查、形式化验证。
  • 第二类是故障检测（Fault Detection）：交付前把故障暴露出来并解决，例如测试、调试、测试驱动开发。
  • 第三类是容错（Fault Tolerance）：运行时出现故障后，设法消除或降低影响，例如冗余、备份、重试。

05 防御性编程工具箱

工具箱：输入检查守边界；断言查内部不变量，尽量一条断言只查一个条件，并避免带副作用表达式。

对应正文

防御性编程做什么
  • 防御性编程的核心是承认程序都会有问题，并且会不断被修改。子程序不应该因为传入错误数据而被破坏，即使这些错误数据来自其他子程序。
  • 最基本的做法是输入检查。所有来自外部的数据都不可信，包括文件、用户输入、网络和其他外部接口。函数的输入参数也要检查是否满足规约的前置条件，例如类型是否一致、参数值是否合法、长度是否符合要求。
  • 断言适合处理“理论上永远不应该发生”的内部错误。C++ 中可以使用 里的 assert 宏。断言为真表示程序运行正常；断言为假表示程序发生错误。每个 assert 最好只检查一个条件，不要把多个条件揉在一起，也不要在断言里写会改变环境的语句，例如 assert(i++ < 100)，因为断言可能只在 debug 阶段生效。
C++ 异常和资源安全
  • C++ 中用 throw 抛出异常，用 try...catch 捕获异常。异常对象的类型决定激活哪个 catch；如果当前函数找不到匹配的 catch，会退出当前函数栈，继续到调用者栈中寻找；如果到达 main 仍没有匹配处理，程序会终止。catch(...) 可以捕获任意类型异常。
  • 标准库也提供了一些异常类。bad_alloc 表示动态内存分配失败，out_of_range 常见于 vector 或 string 的 at 成员函数越界访问，invalid_argument 可以表示参数格式不合法。异常类通常提供 what() 方法，用于返回异常原因。
  • 异常处理会带来资源释放问题。课程中的例子是：Func() 中先 new int 得到 p1，随后调用 div()；如果除数为 0，div() 抛出 invalid_argument 并直接回到 main 的 catch，导致 delete p1 没有执行，于是产生内存泄漏。

隔栏：把错误挡在边界外
  • 如果所有代码都做异常和错误处理，程序会变得臃肿，可读性下降。隔栏（barricade）是一种简化错误处理的设计策略：把程序外部和内部隔离，选定某些接口作为安全区域边界，只在边界处对穿越数据做合法性校验。
  • 隔栏让断言和错误处理有了清晰分工。隔栏外部面对不安全的外部数据，应该使用错误处理技术；隔栏内部的数据已经经过校验，理论上应当正确，因此更适合用断言检查内部不变量。如果隔栏内部仍然出错，就说明程序本身出错。

07 方法与权衡

方法与权衡：防御性编程主要属于 Fault Avoidance；同时要理解 Fault Detection、Fault Tolerance，以及过度防御的成本。

对应正文

需要权衡的地方
  • 防御性编程不是越多越好。开发阶段希望暴露出的错误越多越好，因此会加入很多防御性代码；发布阶段则希望错误尽量不要打断用户使用。
  • 过度防御也会制造问题。如果程序在每一个能想到的地方都做参数检查和错误保护，就会变得臃肿而缓慢。额外防御代码还会增加复杂度，反而可能引入新的错误。更合理的做法，是把检查集中在边界、外部输入、资源所有权和关键不变量上。
小结
  • 健壮性关注的是程序面对异常情况时的表现。正确性保证程序在规约内做对，健壮性要求程序在规约之外仍尽可能可控。理解 Mistake -> Fault(Bug) -> Error -> Failure 这条链，可以帮助我们知道错误是怎样从人的过失变成用户可见失效的；而输入检查、断言、异常处理、智能指针和隔栏，则是在不同位置切断这条链的工具。

  • 健壮性关注的是程序面对异常情况时的表现。正确性保证程序在规约内做对，健壮性要求程序在规约之外仍尽可能可控。理解 Mistake -> Fault(Bug) -> Error -> Failure 这条链，可以帮助我们知道错误是怎样从人的过失变成用户可见失效的；而输入检查、断言、异常处理、智能指针和隔栏，则是在不同位置切断这条链的工具。
