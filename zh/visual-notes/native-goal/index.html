01 总览图

总览：从人工 goal-loop 转向原生 /goal supervisor 的整体架构演化。

对应正文

背景
  • 长时间 agent 任务有一个很具体的问题：一次对话或一次命令很容易完成局部工作，但很难稳定追逐一个大目标。
  • 例如“把一本数学教材整理成可导出的知识地图”不是一个单步任务。它需要持续读取来源、拆分章节、生成节点、补证明、检查依赖关系、修正低质量条目、统计覆盖率，并且在最终导出前保留人工审核入口。任何一个环节过早宣布完成，都会让结果看起来完整、实际却不可靠。
  • 所以最初的做法是造一个 goal-loop：让 supervisor 读 GOAL.md，决定下一轮任务；worker 执行局部工作；reviewer 严格检查；如果没过，就再回到 supervisor。这个结构的价值在于，它把“继续推进”从一次聊天变成了可观察的循环。
第一阶段：人工 goal-loop
  • 最早的 goal-loop 解决的是三个问题。
  • 第一，它把目标写进 GOAL.md，避免每一轮都靠对话记忆。GOAL.md 记录最终目标、边界、质量门槛、停止条件和不可接受状态。agent 可以忘记聊天细节，但不能随意改写目标。
  • 第二，它把长期任务拆成 supervisor、worker、reviewer 三个角色。supervisor 决定下一块工作，worker 只完成局部任务，reviewer 判断结果是否足够好。这比单 agent 从头做到尾更容易发现缺口。

Frame 02

02 旧结构的价值与边界

背景与问题：goal-loop 让长任务可观察，但脚本不该成为默认调度脑。

对应正文

暴露出来的问题
  • 继续使用之后，goal-loop 的边界也变得明显。
  • 最大的问题是：脚本承担了太多“循环大脑”的职责。Python loop 负责唤醒下一轮、组织角色、判断状态、推进到下一步。这样虽然稳定，但也容易把系统做成一个外部调度器，而不是让 agent 自己在目标下做判断。
  • 第二个问题是 reviewer 层很重。固定 reviewer 对严格验收很有价值，但并不是每一轮都需要完整 review。很多时候我们只是需要一个只读审计者判断方向是否漂移、证据是否矛盾、是否出现伪完成风险。把所有检查都做成 reviewer pass/fail，会让系统偏向机械循环。
原生 /goal 改变了什么
  • Codex 的原生 /goal 提供了一个新的基础能力：线程级的持久目标。它不是一次性 prompt，而是让 agent 在会话中持续知道自己正在 pursing 什么目标，并支持 pause、resume、clear 等状态控制。
  • 原来 goal-loop 需要靠外部循环不断提醒 supervisor：“你要继续追这个 GOAL。”现在这件事可以交给原生 /goal。也就是说，root supervisor 本身可以持有长期目标，而不是每一轮都被脚本重新启动。
  • 这个变化不意味着所有工具都消失。GOAL.md、任务卡、handoff、audit 文件仍然重要。区别在于：这些文件不再是脚本循环的输入输出胶水，而是 root supervisor 和其他 agent 共享的工作台。
第二阶段：supervisor + worker 的新结构
  • 这里最重要的是 root supervisor。
  • root supervisor 不只是“下一步调度器”。它负责保持最终目标稳定，维护任务板，判断当前任务应该用哪种执行模式，并在证据不足时停下来验证。它可以直接派一个 worker，也可以把大任务拆给 task supervisor。它还可以决定某一步只需要 audit，不需要写文件。
  • worker 的职责反而更窄。worker 可以使用原生 /goal 固化自己的任务卡目标，但它只负责单张任务卡，不判断整个 sandbox 是否 export-ready。它要留下 WORKER_RESULT.md、CHANGED_FILES.md、REMAINING_BLOCKERS.md、SELF_CHECK....

Frame 03

03 原生 /goal 后的新架构

新结构：原生 /goal 保持目标，root supervisor 按语义组织 worker、audit 和人工判断。

对应正文

为什么取消默认 dispatcher
  • 我们曾经尝试过一个轻量 task dispatcher：它扫描 TASK_INDEX.md 和 tasks/TASK-*.md，发现 ready 任务就启动一个绑定任务卡的 /goal worker。这个方向有吸引力，因为它看上去可以自动派生 worker。
  • 但这个实验也暴露出一个关键风险：脚本只能读状态，不能真正理解任务语义。
  • 如果 TASK_INDEX.md 说某任务 ready，但对应任务卡缺失，或者 ROUND_BRIEF.md 指向另一张任务卡，脚本很难判断这是不是一个应该停下来修复的状态。它会倾向于继续派发，因为它被设计成 dispatch，而不是 supervise。
为什么 root supervisor 需要实验权
  • 另一个关键变化是：root supervisor 不应该只会派任务。它还必须有权在方向不清晰时停下来做验证。
  • 这种验证可以很小：抽样检查几份源文件，重算一个 coverage count，跑一个脚本，比较两种任务拆解方式，或者先验证一个 source-audit 修复策略。它的目的不是完成交付，而是判断下一步怎么做才可靠。
  • 这条规则很重要，因为很多失败不是 worker 执行能力不足，而是任务边界本身不清楚。如果 supervisor 在这种时候继续派 worker，系统会制造更多看起来有进展、实际增加混乱的输出。

Frame 04

04 dispatcher 与 validation

设计取舍：取消默认 dispatcher，方向不清晰时先 validation，再 dispatch。

对应正文

当前形态
  • 当前这套架构可以概括成三层行动结构，加一个独立审计结构。
  • 第一层是 root supervisor。它持有原生 /goal，负责战略选择、任务安排、方向纠偏、验证决策和最终是否进入人工审核。
  • 第二层是 task supervisor。它只在复杂任务中出现，负责一个局部子项目的拆解、worker 协调和结果整合。
对后续 skill 的意义
  • 这次演化最实际的结果是：后续 skill 可以直接走新路径。
  • 新的默认路径不再是“准备一个 goal-loop，然后让脚本循环 supervisor/worker/reviewer”。而是：
  • 这让 skill 的职责更清晰。skill 不需要把所有 loop 规则写进 GOAL.md，也不需要把 dispatcher 当成默认调度器。它应该提供任务包、操作手册、审计说明和一条可直接启动 root supervisor 的命令。
我的反思
  • 这两天最值得记录的，不只是架构从 goal-loop 转向原生 /goal supervisor，而是这个转向暴露了我自己在使用 agent 编排时的一个问题。
  • 在做 agent 编排相关任务时，我因为无法直接看到 VPS 上的具体状态，也因为 skill 是一长段英文，没有逐步确认关键环节。结果是，我给 agent 的上下文越来越多，却没有把最重要的事实先确认清楚。上下文变厚之后，agent 看起来更像是在处理一个复杂系统，但实际上关键判断点可能已经漂移了。
  • 这也导致我做出了一个效果不如原生 /goal 指令的架构。它有 supervisor、worker、reviewer、dispatcher、handoff 等很多部件，看起来很完整，但复杂度本身不能证明有效。回头看，原生 /goal 的优势恰恰在于：它把目标保持能力放在 Codex 原生机制里，不需要我额外构造一个很重的外部循环。

Frame 05

05 反思与后续约束

反思与约束：少而清晰，先证据后行动，关键判断留给人。

对应正文

后续开发约束
  • 之后再做类似开发时，我需要给自己加几条更硬的约束。
  • 第一，每一步关键修改都要显式确认：改了哪些文件、为什么改、预期影响是什么、有没有跑过小规模验证。不能只看 agent 的总结，也不能把“已经生成很多东西”当成进展。
  • 第二，重要架构内容必须亲自把关。尤其是涉及长期任务、agent 编排、自动化调度、VPS 状态和项目约定时，不能把判断权完全交给 agent。agent 可以给方案，但关键事实和关键决策要由我确认。
我还没想明白的地方
  • 原生 /goal 在超长任务中到底能稳定保持多久？task supervisor 是否需要固定 handoff 格式？audit-only 的频率应该由时间决定，还是由文件变化决定？这些还需要继续实验。
  • 另外，我还需要继续想清楚：哪些约束应该写进单个项目的 AGENTS.md，哪些应该变成我个人长期使用 Codex 的通用工作协议。

  • SelfImprovingAgent 的 goal-lab 和 sandbox 生命周期管理；
  • 数学教材知识地图项目里的 coverage、proof、dependency audit；
