02 HTTP 报文与方法

HTTP 基础：方法和报文格式是通用协议层，具体路径由服务器定义。

对应正文

HTTP 是什么
  • HTTP 是一种应用层通信协议。它规定了客户端和服务器之间如何组织请求和响应，比如请求行、请求头、请求体、状态码、响应头、响应体等。
  • 一次 HTTP 通信可以理解为：客户端按照固定格式发出请求，服务器按照固定格式返回响应。格式是通用的，但请求里的具体内容由业务决定。服务器会根据请求方法、路径、查询参数、请求体等信息判断自己是否认识这个请求；如果认识，就执行对应动作；如果不认识，就返回错误状态码。
HTTP 报文结构
  • HTTP 通信里有两类报文：请求报文和响应报文。
  • 请求行说明这次请求使用什么方法、访问哪个路径、使用哪个 HTTP 版本。请求头是对请求的补充说明，比如 Host、Accept、Content-Type、Authorization 等。请求体是客户端提交的数据；GET 请求通常没有请求体，POST、PUT、PATCH 更常见请求体。
  • 状态行里包含状态码。200 表示成功，404 表示路径不存在，405 表示方法不允许，500 表示服务器内部错误。响应头说明返回内容的格式、长度等。响应体是服务器真正返回的数据。

05 HTTPS 推送细节

HTTPS 细节：传输层保护连接，凭据和权限决定远端是否接受更新。

对应正文

用 HTTP/API 理解 git push
  • git push 不是把整个文件夹上传到 GitHub，而是把 GitHub 缺少的 Git 对象发过去，并请求更新远程分支指针。
  • 如果本地 main 指向提交 C，GitHub 上的 main 还停在提交 B，并且 C 是基于 B 的新提交，那么 push 的核心请求就是：把 GitHub 缺少的对象发过去，并请求把远程的 refs/heads/main 从 B 更新到 C。
  • 那么 push 会用 HTTPS 承载 Git 协议。这里 HTTPS 是外层通信通道，Git 协议才是真正表达 push 业务内容的协议。
HTTPS push 里的 HTTP 信息
  • Git over HTTP 有 dumb 和 smart 两种方式。现代 Git 托管平台上的 push 通常可以按 Smart HTTP 理解：HTTP/HTTPS 负责传输，Git 自己的服务名、pkt-line、packfile 负责表达 Git 业务内容。

06 Smart HTTP 与 REST API

概念对比：Smart HTTP 是 Git 协议的 HTTP 传输，REST API 是业务接口风格。

对应正文

和 GitHub REST API 的区别
  • 容易混淆的一点是：GitHub REST API 通常是 api.github.com 上返回 JSON 的接口，例如创建 issue、读取 PR、查询仓库信息等。git push 用的是 Git Smart HTTP 接口。它虽然也可能走 HTTP/HTTPS，但请求体和响应体是 Git 协议格式，不是 JSON。
  • 理解这一点之后，git push 就不再像是一个黑箱命令：它本质上是在一个安全连接里，用 Git 协议告诉远程仓库“我有哪些新对象，以及我希望哪些引用移动到哪里”。
