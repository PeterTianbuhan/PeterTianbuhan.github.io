## 这篇文章真正想留下什么

这篇文章来自一次 Transformer 机器翻译作业，但我不想把它写成“我补了哪些课、哪些地方以前没理解”的流水账。那些基础问题对学习过程有用，却未必值得作为发布版文章长期留下。

我更想保留的是几个现在仍然有价值的判断：

1. Transformer 的主线不是“注意力矩阵很神奇”，而是“怎样构造一个能预测下一个 token 的状态表示”。
2. 人所说的语境不是一袋词，而是带上下文和位置关系的集合；Transformer 不顺序扫描，所以必须把位置信息显式写进表示里。
3. `memory` 这个名字容易把人带回 RNN，但 encoder-decoder Transformer 里的 `memory` 不是一根压缩链，而是一组仍可被访问的位置状态。
4. Q/K/V 的命名不只是助记，它把“如何匹配”和“拿到什么内容”拆开了。
5. 多头 attention 不是给每个 head 单独考试，而是让多套 Q/K/V 合作形成状态，最后一起接受任务 loss 的反向传播。
6. embedding 表本身也是可训练参数；语义相近的词靠近，更多是预测任务长期塑形后的结果，不是直接人工规定。
7. 翻译模型里的 `memory` 不能直接等同于通用大模型里的 user input；decoder-only 大模型把 user/system/history 和已生成前缀放在同一条上下文里。
8. 如果只看成功架构，很容易把 Transformer 的形状当成理所当然。以后我更想回头看失败架构为什么失败。

所以这篇不是一篇 Transformer 教程，而是我把作业里的代码、shape 和生成过程压缩成几个理解支点。

## 状态表示才是主线

我以前很容易顺着 RNN 的直觉去想“模型一定要有一个 memory vector”。RNN 的状态像一根线：当前位置读入 token，更新 hidden state，再把这个 hidden state 传给下一步。这个直觉很自然，因为它像人在从左到右读句子。

Transformer 改掉的不是“需要状态”这件事，而是“状态怎样被组织”。它不再把上下文主要压进一个沿时间滚动的向量，而是保留一整组位置状态：每个 token 都有自己的表示，每个位置都可以通过 attention 从可见上下文里取信息。

我现在更愿意这样概括：

```text
RNN:         当前位置状态 = 当前输入 + 上一步状态
Transformer: 当前位置状态 = 当前 token 从可见上下文里按相关性取回信息后形成的新表示
```

这个视角比“Transformer 不用循环，所以并行”更重要。并行只是计算收益；真正改变理解方式的是：上下文不再必须先压缩成一根线，位置之间可以直接建立内容相关的连接。

这也解释了为什么序列长度变化时，参数不需要跟着变。`W_Q`、`W_K`、`W_V` 这些可训练矩阵主要依赖表示维度，而不是 token 数。输入长短改变时，变大的是运行时生成的 attention 矩阵，不是模型参数本身。这个点如果只看 shape 会显得基础，但它背后的 insight 是：**Transformer 把“可变长度关系”留给运行时计算，把“如何变换表示”留给固定参数学习。**

## 语境不是一袋词，位置必须进入状态

我刚开始自然产生了一个问题：人看句子时是有重点的，同一组词放在不同位置也可能有不同含义。那 Transformer 不像 RNN 一样顺序扫描，它怎样知道词的先后、位置和重点？

这里我觉得“语境”这个词可以拆开看。语境不是一堆词的集合，而是一个带上下文、带位置关系、还会随当前问题改变重点的结构。人在顺序阅读时，位置信息天然包含在扫描过程里：先读到什么、后读到什么，本身就是理解的一部分。

Transformer 的自注意力如果只看 token embedding，本质上会很接近一袋词：同一组 token 换个顺序，模型很难知道顺序变了。因为 attention 本身主要计算“内容之间的相关性”，并不会自动知道第几个词在前、第几个词在后。所以 Transformer 需要把位置信息显式注入表示里，典型做法是：

```text
输入状态 = token embedding + positional encoding
```

这样同一个词出现在不同位置时，进入 attention 的表示已经不同；再经过多层 attention 以后，它还会因为周围上下文不同而形成不同状态。

这也让我重新理解 mask。mask 不是 Transformer 获得位置信息的主要来源；位置主要来自 positional encoding。mask 更像是在规定“当前位置允许看哪些位置”。在 decoder 里，causal mask 的意义是让当前位置不能偷看未来 token，只能基于已经出现的前缀形成状态。也就是说：

```text
positional encoding: 告诉模型每个 token 在哪里
mask: 告诉模型当前位置能看见哪些 token
attention: 决定当前位置应该重点看哪些可见 token
```

所以“人看句子有重点”在 Transformer 里更接近 attention 权重的作用；“句子顺序带来含义差异”则主要靠位置编码先写入状态，再由 attention 在不同上下文里动态放大或弱化。

## embedding 不是静态词典，而是会被任务塑形的参数表

顺着位置编码往前看，token embedding 本身也是一层参数。它不是从 token 到向量的固定字典，而是一张可训练的表：

```text
token id -> embedding vector
```

训练时，loss 并不会直接说“把语义相近的词放近一点”。模型真正接受的监督通常是任务目标，比如机器翻译里预测正确的目标 token，语言模型里预测下一个 token。embedding 表会通过反向传播一起被更新。

所以“语义相近的词在 embedding 空间里更近”更像是一种训练结果，而不是直接目标。两个词如果经常出现在相似上下文里、承担相似功能、对最终预测产生相似影响，它们的 embedding 就可能在训练中被推向相近方向。

但这个“近”也不能只理解成语义近。它可能混合了语法功能、上下文分布和任务需要。比如两个词在翻译任务中经常对应相近输出，或者在语言模型中经常触发相似预测，它们就可能靠近。

更重要的是，Transformer 最终拿去预测的通常不是最初的静态 embedding，而是经过位置编码、attention、多层变换之后形成的上下文状态。同一个 token 的初始 embedding 可以相同，但放在不同句子里，最后的 hidden state 会不同。

## Q/K/V 把匹配和内容拆开

Q/K/V 一开始很像三个方便记忆的名字：`Q` 像我要找什么，`K` 像我拿什么被匹配，`V` 像我真正提供出去的内容。这个说法没错，但如果只停在助记层面，就会漏掉一个设计判断。

更关键的是：attention 不是直接说“当前位置从别的位置拿信息”，而是先把问题拆成两层：

```text
Q 和 K 决定相关性
相关性再作用到 V 上，拿回内容
```

也就是：

```text
A = softmax(QK^T / sqrt(d))
H = A V
```

`QK^T` 产生的是当前输入内部的动态关系矩阵；`V` 才是被聚合的内容。匹配和内容分开以后，同一个 token 可以在不同上下文里以不同方式被访问。对我来说，这是 Q/K/V 真正值得留下的地方：它不是三个“翻译成中文的术语”，而是一种把寻址和取值分离的结构。

## 多头 attention：多套关系视角共同形成状态

如果只看一个 attention head，token 间的交互确实像二元关系：当前位置的 `Q` 和另一个位置的 `K` 算相关性，再用这个权重去取 `V`。我自然会问：如果语义里有三元甚至更多元关系，Transformer 怎么处理？

我现在的理解是：高阶关系通常不是显式写成“三元关系表”，而是在多层、多头和非线性变换中被组合出来。

第一层之后，每个 token 的表示已经不是原来的 token embedding，而是在语义空间里被其它 token 改写过的状态。下一层再做 attention 时，参与交互的已经是这些上下文化状态：

```text
第 0 层：h_i = token embedding + position
第 1 层：h_i' = h_i 从其它 token 拿回信息后的状态
第 2 层：h_i'' = 已带上下文的 h_i' 再和其它已带上下文的 h_j' 交互
```

所以表面上每次还是二元 attention，但第二层以后的“二元关系”已经发生在复合状态之间。一个 token 状态里可能已经带着多个 token 的影响，再和另一个状态交互时，多元关系就被隐式折叠进 hidden state 里。

多头 attention 则是在同一层里并行给模型多套 Q/K/V 参数：

```text
head1 = Attention(Q1, K1, V1)
head2 = Attention(Q2, K2, V2)
head3 = Attention(Q3, K3, V3)
...
concat(head1, head2, head3, ...) -> W_O -> 新状态
```

我的直觉是：既然给了多套参数，就希望它们在相同输入下产生不同关系视角，合作建立状态信息。这可以增加模型表达复杂关系的能力。但标准训练里通常不会给每个 head 单独算 loss。loss 仍然来自最后的任务输出：

```text
logits vs 正确 token
```

然后梯度从最终 loss 一路反传到 `W_O`、每个 head 的输出、每套 Q/K/V 参数。也就是说，每个 head 没有单独的“考试成绩”；它们共同贡献最终状态，最终预测错了或对了，责任通过反向传播分摊回各自参数。

这也说明了一个边界：多头 attention 给了模型分工的机会，但并不保证每个 head 都学到人能命名的功能。它们可能学出不同模式，也可能有冗余。attention 机制更早就存在；多头 attention 是 2017 年 Transformer 论文中被系统化并普及的核心设计，现在已经是现代 Transformer 的基础组件。

## Decoder 的本质是把前缀变成问题

机器翻译作业让我真正看清楚的一点是：Decoder 不是拿“未知的下一个词”做输入。它拿的是已经存在的目标端前缀，然后把这个前缀变成一个问题：

```text
在这个源句 memory 下，
在当前目标前缀之后，
下一个 token 应该是什么？
```

训练时，这个问题可以并行构造出来：

```text
tgt_input: <bos> A group of people
tgt_out:       A group of people <eos>
```

推理时，它就变成自回归循环：

```text
<bos> -> A -> group -> of -> ...
```

我不想在文章里过多记录“为什么要错开一位”这种基础细节。真正有价值的是这个转化：**生成不是从空白处猜词，而是不断把已有前缀重新组织成一个状态，再用这个状态提出下一个 token 的问题。**

## 从状态到词表：最后一步不是语言，而是打分

一次完整预测可以压缩成这条链：

```text
源句 -> Encoder -> memory
目标前缀 + memory -> Decoder -> H
最后位置状态 -> generator -> logits
logits -> 采样或取最大值 -> next token
```

这里 `H` 不是句子，也不是答案；它是一组状态表示。推理时真正用来预测下一步的是最后位置的状态：

```text
h = H[-1]
logits[-1] = h W + b
```

这个接口对我很重要。它提醒我：Transformer 的大部分结构都在服务于“产生一个好的状态表示”，而最终输出仍然是对词表的打分。所谓“模型会说话”，落到这一层其实是：

```text
当前状态 -> 整个词表上的分数分布 -> 选出下一个 token
```

temperature 也应该放在这个位置理解。它不是训练时帮模型跳出局部最优，而是在推理时改变 logits 变成采样分布的形状：温度低，分布更尖锐；温度高，分布更平。这个修正比“temperature 让模型更有创造力”更值得保留，因为它把直觉放回了正确的数据流位置。

## `memory` 这个名字反而容易误导

回到作业代码，`memory` 这个变量名很容易让我把 Transformer 又想回 RNN。它们确实都承载上下文，但相似性到这里就该停住。

RNN 的 memory 更像顺序压缩出来的一条链；encoder-decoder Transformer 里的 `memory` 是源句每个位置经过 Encoder 之后得到的一组状态：

```text
memory: [S, B, E]
```

这组状态没有先被压成一个固定长度摘要。Decoder 可以通过 cross-attention 根据当前目标前缀动态决定该看源句里的哪些位置。

我之前用过“平权”这个词，但它容易被误解成每个位置权重相等。更准确地说是：**源句的每个位置都保留了被访问的资格，具体看谁、看多少，由 Decoder 当前状态动态决定。**

这个理解也解释了为什么 Transformer 适合处理长距离关系。不是因为它“天然理解长句”，而是因为远处 token 不必沿着一根递归链一步步传过来；它可以在 attention 里被直接寻址。当然，这会换来 attention 矩阵的平方级计算成本。

## 翻译模型和通用大模型的类比边界

上面的理解首先来自机器翻译的 encoder-decoder 结构：

```text
源语言句子 -> Encoder -> memory
目标语言前缀 + memory -> Decoder -> 下一个目标语言 token
```

但通用聊天大模型通常不是把 user input 单独编码成一个 `memory`，再让 Decoder 去读取。很多通用大模型是 decoder-only 结构：

```text
system prompt + user input + assistant 已生成前缀
-> causal self-attention
-> 最后位置状态
-> logits
-> 下一个 token
```

所以二者可以类比，但不能一一替换：

```text
翻译模型里的源句 memory
≈ decoder-only 大模型里，当前位置之前可被注意到的上下文 token
```

区别在于，机器翻译有单独的 Encoder 输出和 cross-attention；decoder-only 大模型把 user/system/history 和 assistant 前缀都放在同一条序列里，通过 causal self-attention 组织状态。

这也让我重新看“人说话”和 Transformer 的关系。人的说话也会被前文、当前意图和对方输入约束，但模型不是先拥有一个完整想法再翻译成文字；更基本的机制是不断用已有上下文形成当前位置状态，再预测下一个 token。这个类比有启发，但不能过度拟人化。

## 代码只作为校准

作业代码对我有用的地方，是把上面的判断压回到具体数据流里：

| Tensor | shape | 我需要它帮我校准什么 |
| --- | --- | --- |
| `src` | `[S, B]` | 源语言 token id 序列 |
| `tgt_input` | `[T, B]` | 输入 Decoder 的目标语言前缀 |
| `memory` | `[S, B, E]` | Encoder 输出的源句位置状态 |
| `outs` | `[T, B, E]` | Decoder 形成的目标端状态 |
| `logits` | `[T, B, V_tgt]` | 每个目标位置对词表的打分 |
| `tgt_out` | `[T, B]` | 训练时错开一位的监督标签 |

`evaluate()` 和 `translate()` 的区别也可以这样压缩：前者有完整目标句，可以并行算 loss；后者没有答案，只能用已经生成的前缀一步步往后走。

单层结构上，Encoder 做源语言内部 self-attention；Decoder 先做目标前缀的 masked self-attention，再通过 cross-attention 读 `memory`。这就够了。发布文章里不需要把所有代码细节都铺开，否则主线会从 insight 退回作业说明。

## 我不想提前写成结论的部分

作业里还有 `<eos>`、embedding 最近邻、temperature、Tiny/Base/Wide 超参数实验。这里我现在只愿意保留边界，不抢写结论。

`<eos>` 的价值，不是“序列需要一个结束符”这么浅，而是它让训练目标里出现“现在应该停止”的监督信号。temperature 的价值，也不是训练阶段跳出局部最优，而是推理阶段改变 logits 到采样分布的转换。Tiny/Base/Wide 的训练还没完成，就不应该先把结果写成漂亮结论。

我更想把后续问题留给另一篇文章：Transformer 之前一定有很多失败或半成功的架构尝试。只看成功者，容易把成功架构的每个设计都解释得很合理；回头看失败者为什么失败，可能更能看清楚 Transformer 到底解决了什么，又把什么问题留给了后来的模型。
