## 背景

Git 是一种分布式版本控制系统。它的核心作用不是“把代码传到 GitHub”，而是记录项目在不同时间点的变化，让修改可以被追踪、解释、审查和回退。

当项目逐渐变复杂时，版本控制可以帮助回答几个基础问题：

- 当前项目处在什么状态？
- 这次改动具体改变了什么？
- 如果出现问题，能不能回到之前稳定的版本？
- 哪些改动已经被确认，哪些还只是实验？
- 多个任务并行时，怎样避免互相覆盖？

在有 agent 参与开发的工作流里，这些问题会变得更重要。因为 agent 可以快速产生改动，但项目仍然需要清楚的边界、审查和恢复能力。

## 事实记录

### 工作区、暂存区和提交

Git 的基本使用可以先理解成三层：

第一层是工作区，也就是正在编辑的真实文件。这里可以试错，也可以临时探索。

第二层是暂存区。暂存区用于从一组改动里挑出准备进入下一次提交的部分。

第三层是 commit。commit 是一次带有意图的项目状态记录。一个清楚的 commit 应该服务于同一个目的，并能说明这次为什么改、改了什么。

因此，`git status` 和 `git diff` 是提交前非常重要的检查：

- `git status` 用来确认哪些文件被修改、添加或删除。
- `git diff` 用来查看具体改动内容。
- `git add` 用来选择哪些改动进入暂存区。
- `git commit` 用来保存一组有意义的变化。

### 分支

分支用于隔离风险。主分支通常应该保持相对稳定，实验性修改、功能开发和任务执行可以放在独立分支里。

常见做法是：

- 小修复使用短分支。
- 页面重建或功能开发使用独立功能分支。
- agent 自动执行任务时，为每个 Linear issue 创建独立分支。

这样可以让每个任务都有自己的工作空间，避免不同任务的改动混在一起。

### Pull Request

Pull Request 是代码进入主线之前的审查缓冲区。它把“代码已经改完”和“代码可以合并”分开。

在 agent 工作流中，PR 尤其重要：

- agent 可以完成修改、运行验证、创建 draft PR。
- reviewer 或人类可以检查目的是否达成。
- 合并和发布可以保持人工确认。

这能降低 agent 直接把不完整或有风险的修改推入主线的概率。

### 回退

版本控制也提供恢复能力。项目出问题时，关键不是永远不犯错，而是能定位是哪次变化引入了问题，并在必要时回退。

为了让回退更容易，每个 commit 应该尽量聚焦：

- 不把无关修改塞进同一个 commit。
- 高风险修改走分支和 PR。
- 发布前检查构建产物、域名、目标分支和线上页面。

### 和当前工作流的关系

Native Symphony 的目标是把自然语言任务转成可执行、可审查、可复盘的工作块。Git 在这里承担的是项目变化的轨道：

- Linear issue 记录任务目的。
- Git branch 隔离执行空间。
- commit 记录具体变化。
- PR 提供验收缓冲区。
- Linear 评论记录结果和证据。

因此，一个理想的任务应该让 Linear issue、Git branch、commit 和 PR 能互相对应。
