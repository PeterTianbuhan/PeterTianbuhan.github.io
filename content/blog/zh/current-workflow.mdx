## 现在这套工作流是什么样子

目前我想要的工作方式，可以先概括成一句话：我只参与两个环节，一开始把模糊想法说清楚，最后验收产品和提出意见；中间的任务发布、代码修改、审查、PR、部署和复盘，都尽量交给一套可观察的 agent 工作流。

现在的闭环大致是这样：

1. 我先和 Codex 讨论一个想法，把它从“我想做点什么”整理成一个清晰目标。
2. Codex 帮我写成 Linear 任务，里面包含目标、工作目录、验收标准和要避免的负面影响。
3. VPS 上常驻的 worker 定期读取 Linear，领取 `Todo` 状态的任务。
4. worker 在对应 GitHub 仓库里拉取代码、创建分支，然后调用 Codex CLI 完成实现。
5. 实现结束后，另一个 reviewer agent 根据任务目的和证据判断是否达成目标。
6. 只有 reviewer 通过，worker 才提交代码、推送分支、创建 draft PR，并把结果写回 Linear。
7. 我最后查看 PR、预览或线上结果，决定是否合并或继续修改。
8. 每天或每个阶段，再根据 Linear、日志和 PR 总结这套流程哪里做得好、哪里还不可靠。

这不是为了把所有决定都交给 AI。相反，我越来越觉得这套系统的核心价值，是让 agent 的工作变得有边界、有记录、有验收，也能从失败中留下可以改进的痕迹。

## 最初的动机

一开始我关注的是 OpenAI 的 Symphony 项目。吸引我的不是某个具体技术栈，而是它背后的工作模式：任务被发布出来，智能体可以自己领取、执行、回写结果。这个模式很像一个小型软件团队，只不过执行者是 agent。

我当时最想解决的问题并不是“让 Codex 帮我写一次代码”。直接在对话框里让 Codex 做事已经很方便了。真正的问题是，当项目变多之后，单个对话框很难承担项目管理的职责。

我希望看到：

- 每个项目现在有哪些任务；
- 哪些任务正在被处理；
- 哪些任务失败了，失败原因是什么；
- Codex 经常在哪些地方做得不好；
- 哪些规则应该沉淀进下一版 worker、skill 或任务模板；
- 一段时间以后，我的项目推进是否真的更稳定了。

所以 Linear 对我来说不是一个普通的待办列表，而是 agent 工作的外部项目层。Codex 负责执行，Linear 负责记录任务、状态和结果。

## 从 Linear 开始

第一步是把 Linear 接起来。为了让 worker 能读写任务，我配置了 Linear API Key，并找到项目 slug。这个阶段让我意识到一个很重要的边界：API Key 和 slug 不是“智能”的部分，它们只是让系统知道应该进入哪个 workspace、哪个 project，以及用什么身份读写 issue。

之后我加了一个 `create-task` 功能。它让我可以在对话里和 Codex 讨论任务，再由 Codex 把任务发布到 Linear。这个小功能很关键，因为它把“讨论想法”和“发布任务”连到了一起。

但很快我发现，光有任务标题和描述还不够。一个模糊任务进入系统以后，后面的 worker 很容易误解范围。于是任务发布逐渐变成了一个更严格的过程：先说明真实目的，再说明用户应该看到什么变化，再说明哪些副作用不可接受，最后列出 reviewer 可以检查的证据。

这就是现在的 Goal Spec。它不是复杂模板，而是一个提醒：验收首先看目的有没有达成，其次看有没有造成额外负面影响，最后才看命令、diff、截图、PR 或线上链接这些证据。

## Windows 本地版本的局限

最早的 worker 是在 Windows 本机上跑的。它可以读取 Linear，调用 Codex CLI，修改本地项目，再把结果写回 Linear。这个最小闭环证明了方向是可行的。

但它也很快暴露出问题。Windows 上路径、权限、后台进程、终端可见性都比较脆弱；更重要的是，我的电脑关机或休眠以后，自动化就停止了。这和我希望的“后台持续推进项目”不一致。

这时我开始考虑把 worker 迁移到 VPS。VPS 的角色不是托管网站，而是作为常驻执行节点。它负责定时检查 Linear、拉取 GitHub 仓库、调用 Codex CLI、创建 PR、写回结果和生成复盘。

这个决定把工作流从“本机实验”推进到了“可以长期运行的控制节点”。

## 迁移到 VPS

VPS 迁移过程中，我做了几件基础但重要的事情。

首先是安全边界。worker 不应该长期用 root 身份运行，所以 VPS 上创建了普通用户 `symphony`。它只负责运行 Native Symphony worker 和保存工作区。

其次是运行环境。VPS 上需要 Git、Python、Node、npm、GitHub CLI 和 Codex CLI。Codex CLI 通过登录后可以在服务器上执行任务，GitHub CLI 负责 clone、push 和创建 PR。

再次是定时运行。原来依赖本机 Codex App 自动化，迁移后改成 systemd timer。现在 VPS 每五分钟执行一次 `poll --once --quiet-idle`。如果没有任务，它保持安静；如果有任务，它领取一个任务并进入完整工作块。

这个设计让我更接近理想状态：我不用一直盯着终端，也不用手动启动每个任务。任务进入 Linear 以后，VPS 会定期检查并处理。

## 从直接改代码到 GitHub PR

刚开始 worker 只需要完成 Linear -> Codex -> Linear：读取任务、执行、写回结果。但真正用于项目开发时，这还不够。代码应该进入 GitHub，应该有分支、commit、PR 和审查入口。

于是下一步是把流程升级成：

```text
Linear -> VPS worker -> Codex CLI -> GitHub branch -> draft PR -> Linear
```

这一步打通以后，Linear issue 和 GitHub PR 可以互相对应。worker 不再只是“在某个目录里改了文件”，而是能把结果变成一个可审查的 PR。对我来说，这很重要，因为 PR 是产品验收前的缓冲区：它允许 agent 做完工作，但不直接把结果合并到线上。

默认策略也因此变得清楚：agent 可以创建 PR，但不自动合并；最终合并和发布仍然需要我验收。

## HomePage 是第一次真实证明

这套工作流真正让我觉得有效，是因为它成功推动了我的个人网站改版。

一开始我对网站的感觉是：之前的前端风格太像黑客终端和深色科技面板，不适合我想要的个人站。我想把它推倒重建成浅色纸面、学术笔记型的首页。这个目标先在对话里被澄清，再发布到 Linear，然后由 worker 在 VPS 上执行。

过程中任务被拆成了几步：

- 先让源码分支适合 worker PR；
- 再实现新的浅色学术笔记风格首页；
- 然后修复发布管线，保证 GitHub Pages 需要的是静态产物；
- 最后把生成后的静态 bundle 发布到 `main`，并验证自定义域名不再 404。

这次成功让我第一次看到完整闭环的价值。不是单次生成代码，而是从产品想法、任务拆解、后台执行、PR 审查、线上验证到复盘，形成了一条可以重复使用的路径。

网站恢复正常以后，我意识到：这套系统确实可以把自然语言想法推进到真实产品变化。

## 复盘机制比自动执行更重要

我现在越来越确信，真正值得做的不是“让 agent 更自由”，而是让 agent 的每次行动都能被复盘。

这套工作流会保存 progress、run summary、acceptance review 和 Linear 评论。每天的 review 可以汇总过去一段时间的任务结果，并按照一些缺陷类型分类，例如：

- `task_unclear`：任务本身不够清楚；
- `env_or_permission`：环境、权限、登录、API key 等问题；
- `implementation_failed`：实现失败；
- `validation_missing`：验证不足；
- `scope_drift`：范围漂移；
- `git_or_publish`：Git、PR 或发布问题。

这些分类不是为了给 agent 打分，而是为了知道系统应该怎么进化。比如这次 HomePage 流程暴露出几个具体问题：

- 失败运行可能留下脏工作区，影响下一次重试；
- 前端任务只看 lint/build 还不够，应该有截图或浏览器证据；
- “发布网站”这种任务必须区分“修发布管线”和“真的更新线上 main”；
- PR 合并后的 live verification 还需要更清晰地纳入流程。

这些问题后来都变成了新的 Linear 任务。也就是说，失败和摩擦不再只是一次对话里的抱怨，而是会进入下一轮系统改进。

## `/goal` 给我的启发

最近我还看到了 Codex CLI 的 `/goal` 功能。它的方向很符合我的直觉：让 agent 记住一个目标，并持续运行直到目标完成或达到预算限制。

不过我现在不会把它直接放进正式 worker。原因很简单：它目前还是一个偏实验性的功能，需要开启 feature flag，也需要在 sandbox 中观察 pause、resume、complete 等行为是否稳定。

所以我现在的做法是把 `/goal` 放进 goal lab。正式工作流仍然依赖 Linear、worker、reviewer 和 PR；`/goal` 先作为一个实验对象，用低风险 sandbox 任务验证。如果它足够稳定，未来它可能成为 worker 内部长任务执行的一部分。

这也符合我对这套系统的原则：新能力可以引入，但不能让正在工作的主流程失去可控性。

## 我希望自己最终怎样使用它

理想状态下，我每天的使用方式会很简单：

1. 我用自然语言说出一个想法；
2. Codex 帮我把它澄清成 Goal Spec 和 Linear 任务；
3. VPS worker 后台领取任务；
4. Codex CLI 在对应 repo 中完成实现；
5. reviewer agent 根据目的和证据验收；
6. GitHub PR 出现，Linear 收到结果；
7. 我验收产品，决定合并、修改或继续拆任务；
8. 系统把这次过程写入复盘，形成下一轮改进。

如果这个流程稳定下来，很多项目推进会变得轻很多。我的角色会更像产品负责人和最终验收者，而不是每一步都亲自操作的人。

但我也不想把它说成已经完成。现在它还在很早期：视觉验收、失败隔离、发布后验证、任务风险分级、`/goal` 沙盒测试都还需要继续打磨。它目前最有价值的地方，是已经证明了方向：自然语言想法可以通过一套可观察、可复盘的 agent 工作流，变成真实网站上的变化。

这篇文章就是这个阶段的第一条记录。
