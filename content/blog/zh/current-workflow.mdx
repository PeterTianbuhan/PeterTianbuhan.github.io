## 现在这套工作流是什么样

目前我希望形成的工作方式，可以概括成一句话：

我只重点参与两个环节：一开始把模糊想法说清楚，最后验收产品和提出意见；中间的任务发布、代码修改、审查、PR、部署和复盘，都尽量交给一套可观察的 agent 工作流。

现在的闭环大致是：

1. 我先和 Codex 讨论一个想法，把它从“我想做点什么”整理成清晰目标。
2. Codex 帮我写成 Linear 任务，里面包含目标、工作目录、验收标准和需要避免的负面影响。
3. VPS 上常驻的 worker 定期读取 Linear，领取 `Todo` 状态的任务。
4. worker 在对应 GitHub 仓库中拉取代码、创建分支，然后调用 Codex CLI 完成实现。
5. 实现结束后，reviewer agent 根据任务目的和证据判断是否达成目标。
6. 只有 reviewer 通过，worker 才提交代码、推送分支、创建 draft PR，并把结果写回 Linear。
7. 我最后查看 PR、预览或线上结果，决定是否合并、继续 repair，或者拆出后续任务。
8. 每天或每个阶段，再根据 Linear、日志、PR 和 review 总结这套流程哪里稳定，哪里还需要改进。

这不是为了把所有决定都交给 AI。相反，它的价值在于让 agent 的工作有边界、有记录、有验收，也能从失败中留下可以改进的痕迹。

## 最初的动机

一开始吸引我的是 OpenAI Symphony 项目里的工作模式：任务被发布出来，智能体可以自己领取、执行、回写结果。

这件事真正吸引我的地方不是某个具体技术栈，而是一种项目管理方式。直接在对话框里让 Codex 帮我做一次事已经很方便，但当项目变多之后，单个对话框很难承担项目管理职责。

我希望看到：

- 每个项目现在有哪些任务；
- 哪些任务正在被处理；
- 哪些任务失败了，失败原因是什么；
- Codex 经常在哪些地方做得不好；
- 哪些规则应该沉淀进下一版 worker、skill 或任务模板；
- 一段时间以后，项目推进是否真的更稳定。

所以 Linear 对我来说不是一个普通待办列表，而是 agent 工作的外部项目层。Codex 负责执行，Linear 负责记录任务、状态和结果。

## 从 Linear 开始

第一步是把 Linear 接起来。为了让 worker 能读写任务，我配置了 Linear API Key，并找到项目 slug。这个阶段让我意识到一个边界：API Key 和 slug 不是“智能”的部分，它们只是让系统知道应该进入哪个 workspace、哪个 project，以及用什么身份读写 issue。

之后我加了 `create-task` 能力。它让我可以在对话里和 Codex 讨论任务，再由 Codex 把任务发布到 Linear。这个小功能很关键，因为它把“讨论想法”和“发布任务”连到了一起。

但很快我发现，只有任务标题和描述还不够。一个模糊任务进入系统以后，后面的 worker 很容易误解范围。于是任务发布逐渐变成一个更严格的过程：

- 说明真实目的；
- 说明用户完成后应该看到什么变化；
- 说明哪些副作用不可接受；
- 说明 reviewer 可以检查哪些证据。

这就是现在的 Goal Spec。它不是复杂模板，而是一种提醒：验收首先看目的有没有达成，其次看有没有造成额外负面影响，最后才看命令、diff、截图、PR 或线上链接这些证据。

## Windows 本地版本的局限

最早的 worker 是在 Windows 本机上跑的。它可以读取 Linear、调用 Codex CLI、修改本地项目，再把结果写回 Linear。这个最小闭环证明方向可行。

但它也很快暴露出问题。Windows 上路径、权限、后台进程、终端可见性都比较脆弱；更重要的是，我的电脑关机或休眠以后，自动化就停止了。这和我希望的“后台持续推进项目”不一致。

于是我开始考虑把 worker 迁移到 VPS。VPS 的角色不是托管网站，而是作为常驻执行节点。它负责定时检查 Linear、拉取 GitHub 仓库、调用 Codex CLI、创建 PR、写回结果和生成复盘。

这个决定把工作流从“本机实验”推进到了“可以长期运行的控制节点”。

## 迁移到 VPS

VPS 迁移过程中，我做了几件基础但重要的事情。

首先是安全边界。worker 不应该长期用 root 身份运行，所以 VPS 上创建了普通用户 `symphony`。它只负责运行 Native Symphony worker 和保存工作区。

其次是运行环境。VPS 上需要 Git、Python、Node、npm、GitHub CLI 和 Codex CLI。Codex CLI 登录后可以在服务器上执行任务，GitHub CLI 负责 clone、push 和创建 PR。

再其次是定时运行。原来依赖本机 Codex App 自动化，迁移后改成 systemd timer。现在 VPS 可以定期执行 `poll --once --quiet-idle`。如果没有任务，它保持安静；如果有任务，它领取一个任务并进入完整工作块。

这个设计让我更接近理想状态：我不用一直盯着终端，也不用手动启动每个任务。任务进入 Linear 以后，VPS 会定期检查并处理。

## 从直接改代码到 GitHub PR

刚开始 worker 只需要完成 Linear -> Codex -> Linear：读取任务、执行、写回结果。但真正用于项目开发时，这还不够。代码应该进入 GitHub，应该有分支、commit、PR 和审查入口。

于是流程升级成：

```text
Linear -> VPS worker -> Codex CLI -> GitHub branch -> draft PR -> Linear
```

这一步打通以后，Linear issue 和 GitHub PR 可以互相对应。worker 不再只是“在某个目录里改了文件”，而是能把结果变成一个可审查的 PR。

默认策略也因此变得清楚：agent 可以创建 PR，但不自动合并；最终合并和发布仍然需要人工验收。

## HomePage 是第一次真实证明

这套工作流真正让我觉得有效，是因为它成功推动了我的个人网站改版。

一开始我对网站的感觉是：之前的前端风格太像黑客终端和深色科技面板，不适合我想要的个人站。我想把它推倒重建成浅色纸面、学术笔记型首页。

这个目标先在对话里被澄清，再发布到 Linear，然后由 worker 在 VPS 上执行。过程中任务被拆成几步：

- 让源码分支适合 worker PR；
- 实现新的浅色学术笔记风格首页；
- 修复发布管线，保证 GitHub Pages 需要的是静态产物；
- 把生成后的静态 bundle 发布到 `main`，并验证自定义域名不再 404。

这次成功让我第一次看到完整闭环的价值。不是单次生成代码，而是从产品想法、任务拆解、后台执行、PR 审查、线上验证到复盘，形成了一条可以重复使用的路径。

网站恢复正常以后，我意识到：这套系统确实可以把自然语言想法推进到真实网站上的变化。

## 学习记录进入 Obsidian vault

在工作流逐渐跑通以后，我又遇到另一个问题：项目推进会产生很多过程性记录、学习笔记和阶段性理解。如果这些内容只留在对话里，它们很快会散掉；如果直接放到网站，又会过早公开，混入不成熟的想法。

所以我把之前的 `my-cognitive-vault` 仓库重建成了一个 Obsidian 兼容的学习笔记仓库。旧的数学笔记不是由真实学习过程产生的，所以被删除；仓库保留下来，重新作为真实学习记录的源头。

这个 vault 符合当前工作流的方向，因为它负责承接“学习和记录”：

- 项目推进中产生的事实记录；
- 学习过程中出现的概念、问题和理解；
- Codex 帮我整理的事实层、时间线和结构；
- 我自己后续补充的“我的理解”；
- 未来可能整理成公开文章的候选稿。

现在新的内容流是：

```text
真实学习 / 项目过程
-> my-cognitive-vault
-> 整理成 Obsidian 笔记
-> 判断是否 public candidate
-> 候选内容进入 HomePage
-> 网站只展示公开版本
```

这个变化很重要。它让学习和记录先回到私人 vault 中生长，而不是一开始就进入公开表达。网站可以继续作为最终展示面，但它不应该承担真实学习回顾的主要职责。

## 笔记流的默认规则

从这一步开始，笔记内容的流向应该优先进入 `my-cognitive-vault`。

默认规则是：

- 新学习、新项目、新想法，先进入 vault；
- 默认为 `public: false`；
- Codex 可以负责事实层、结构、时间线、链接和元数据；
- “我的理解”保留给我补充；
- 只有我明确说“可以发布”或“整理成文章”，才进入 HomePage 流程。

这个规则让笔记系统和网站系统分开。vault 可以不完美，可以有草稿、困惑和失败尝试；网站则只展示经过选择的内容。

## 复盘机制比自动执行更重要

我现在越来越确信，真正值得做的不是“让 agent 更自由”，而是让 agent 的每次行动都能被复盘。

这套工作流会保存 progress、run summary、acceptance review 和 Linear 评论。每天的 review 可以汇总过去一段时间的任务结果，并按缺陷类型分类，例如：

- `task_unclear`：任务本身不够清楚；
- `env_or_permission`：环境、权限、登录、API key 等问题；
- `implementation_failed`：实现失败；
- `validation_missing`：验证不足；
- `scope_drift`：范围漂移；
- `git_or_publish`：Git、PR 或发布问题。

这些分类不是为了给 agent 打分，而是为了知道系统应该怎么进化。

比如 HomePage 流程暴露出几个具体问题：

- 失败运行可能留下脏工作区，影响下一次重试；
- 前端任务只看 lint/build 还不够，应该有截图或浏览器证据；
- “发布网站”这种任务必须区分“修发布管线”和“真的更新线上 main”；
- PR 合并后的 live verification 还需要更清楚地纳入流程；
- Linear 输出需要尽量中文化，减少人工阅读负担。

这些问题后来都可以变成新的 Linear 任务。也就是说，失败和摩擦不再只是一次对话里的抱怨，而是会进入下一轮系统改进。

## 长期记忆不应该自动堆积

在反复使用 agent 以后，还有一个问题会变得明显：如果每次运行都把日志直接写进长期记忆，系统很快会堆满噪声。

所以现在我更倾向于三层记忆：

- 短期：`logs/runs`、progress、PR、Linear 评论，用来还原每次运行。
- 中期：daily review，总结最近一段时间的任务、失败模式和改进建议。
- 长期：只有经过人工审核、能改变未来行为的规则，才进入 skill、worker rules 或 reviewer prompt。

也就是说，raw logs 是证据，不是记忆；daily review 是复盘，不是规则；长期记忆候选必须有重复证据或高影响事故支撑。

这个原则能避免“垃圾记忆堆积”。我不希望系统越用越乱，而是希望它从真实失败中提取少量稳定规则。

## `/goal` 给我的启发

最近我还看到 Codex CLI 的 `/goal` 功能。它的方向很符合我的直觉：让 agent 记住一个目标，并持续运行直到目标完成或达到预算限制。

不过我现在不会把它直接放进正式 worker。原因很简单：它仍然需要在 sandbox 中观察 pause、resume、complete 等行为是否稳定。

所以现在的做法是把 `/goal` 放进 goal lab。正式工作流仍然依赖 Linear、worker、reviewer 和 PR；`/goal` 先作为实验对象，用低风险 sandbox 任务验证。如果它足够稳定，未来可能成为 worker 内部长任务执行的一部分。

这也符合我对这套系统的原则：新能力可以引入，但不能让正在工作的主流程失去可控性。

## 我希望自己最终怎样使用它

理想状态下，我每天的使用方式会很简单：

1. 我用自然语言说出一个想法；
2. Codex 帮我把它澄清成 Goal Spec 和 Linear 任务；
3. VPS worker 后台领取任务；
4. Codex CLI 在对应 repo 中完成实现；
5. reviewer agent 根据目的和证据验收；
6. GitHub PR 出现，Linear 收到结果；
7. 我验收产品，决定合并、修改或继续拆任务；
8. 系统把这次过程写入复盘，形成下一轮改进；
9. 如果这次过程产生了值得保留的学习内容，先进入 `my-cognitive-vault`；
10. 如果某篇笔记适合公开，再整理成 HomePage 文章。

这样一来，项目推进和学习记录就不再是两套互相割裂的东西。项目会产生笔记，笔记会沉淀理解，成熟内容再进入网站。

## 我的理解

一套好的工作流应该高效，高质，可进化。依托codex强大的任务执行能力，只需要用清晰可执行的指令描述需求，就能在较短的时间内拿到成果。所以，目前最需要提升的部分就是如何定义和达到清晰，这是这套工作流需要长期努力的方向。一个同样值得努力的方向是降低实现过程所消耗的精力，把中间过程全权交付给codex，把思考留给行动前，让思维能自由翱翔。另一方面，项目和学习都需要留下痕迹，否则很容易陷入空想，多回顾和反思也能促成进步。
