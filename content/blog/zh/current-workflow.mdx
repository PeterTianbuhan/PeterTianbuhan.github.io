## 现在这套工作流是什么样

我现在希望形成的工作方式，可以概括成一句话：

我只重点参与两个环节：一开始把模糊想法说清楚，最后验收产品和提出判断。中间的任务发布、代码修改、审查、PR、发布、监控和复盘，尽量交给一套可观察、可汇报的 agent 工作流。

当前闭环大致是：

1. 我先和 Codex 讨论一个想法，把它从“我想做点什么”整理成清晰目标。
2. Codex 帮我写成 Linear 任务，里面包含目标、工作目录、验收标准和需要避免的负面影响。
3. worker 定期读取 Linear，领取 `Todo` 状态的任务。
4. worker 在对应 GitHub 仓库中拉取代码、创建分支，然后调用 Codex CLI 完成实现。
5. 实现结束后，reviewer agent 根据任务目的和证据判断是否达成目标。
6. 只有 reviewer 通过，worker 才提交代码、推送分支、创建 draft PR，并把结果写回 Linear。
7. 我最后查看 PR、预览或线上结果，决定是否合并、继续 repair，或者拆出后续任务。
8. 如果任务很长，独立的监控 Codex 读取精简状态，把关键变化用短消息发回对话，而不是让我一直盯着终端。
9. 每天或每个阶段，再根据 Linear、日志、PR 和 review 总结这套流程哪里稳定，哪里还需要改进。

这不是为了把所有决定都交给 AI。相反，它的价值在于让 agent 的工作有边界、有记录、有验收，也能从失败中留下可以改进的痕迹。

## 最初的动机

一开始吸引我的是 OpenAI Symphony 项目里的工作模式：任务被发布出来，智能体可以自己领取、执行、回写结果。

这件事真正吸引我的地方不是某一个具体技术栈，而是一种项目管理方式。直接在对话框里让 Codex 帮我做一次事已经很方便，但当项目变多之后，单个对话框很难承担项目管理职责。

我希望看到：

- 每个项目现在有哪些任务；
- 哪些任务正在被处理；
- 哪些任务失败了，失败原因是什么；
- Codex 经常在哪些地方做得不好；
- 哪些规则应该沉淀进下一版 worker、skill 或任务模板；
- 一段时间以后，项目推进是否真的更稳定。

所以 Linear 对我来说不是一个普通待办列表，而是 agent 工作的外部项目层。Codex 负责执行，Linear 负责记录任务、状态和结果。

## 从 Linear 到 Goal Spec

第一步是把 Linear 接起来。为了让 worker 能读写任务，我配置了 Linear API Key，并找到项目 slug。这个阶段让我意识到一个边界：API Key 和 slug 不是“智能”的部分，它们只是让系统知道应该进入哪个 workspace、哪个 project，以及用什么身份读写 issue。

之后我加入了 `create-task` 能力。它让我可以在对话里和 Codex 讨论任务，再由 Codex 把任务发布到 Linear。这一小步很关键，因为它把“讨论想法”和“发布任务”连到了一起。

但很快我发现，只有任务标题和描述还不够。一个模糊任务进入系统以后，后面的 worker 很容易误解范围。于是任务发布逐渐变成一个更严格的过程：

- 说明真实目的；
- 说明用户完成后应该看到什么变化；
- 说明哪些副作用不可接受；
- 说明 reviewer 可以检查哪些证据；
- 说明目标 workspace、仓库、风险等级和发布策略。

这就是现在的 Goal Spec。它不是复杂模板，而是一种提醒：验收首先看目的有没有达成，其次看有没有造成额外负面影响，最后才看命令、diff、截图、PR 或线上链接这些证据。

## VPS worker 和 GitHub PR

最早的 worker 是在 Windows 本机上跑的。它可以读取 Linear、调用 Codex CLI、修改本地项目，再把结果写回 Linear。这个最小闭环证明方向可行。

但本机运行有明显限制。Windows 上路径、权限、后台进程和终端可见性都比较脆弱；更重要的是，我的电脑关机或休眠以后，自动化就停止了。这和我希望的“后台持续推进项目”不一致。

于是我开始把 worker 迁移到 VPS。VPS 的角色不是托管网站，而是作为常驻执行节点。它负责定时检查 Linear、拉取 GitHub 仓库、调用 Codex CLI、创建 PR、写回结果和生成复盘。

真正用于项目开发时，流程也从简单的：

```text
Linear -> Codex -> Linear
```

升级成：

```text
Linear -> VPS worker -> Codex CLI -> GitHub branch -> draft PR -> Linear
```

这一步打通以后，Linear issue 和 GitHub PR 可以互相对应。worker 不再只是“在某个目录里改了文件”，而是能把结果变成一个可审查的 PR。

默认策略也因此更清楚：agent 可以创建 PR，但不自动合并；最终合并和发布仍然需要人工验收。

## HomePage 是第一批真实验证

这套工作流真正让我觉得有效，是因为它成功推动了我的个人网站改版。

一开始我对网站的感觉是：之前的前端风格太像深色科技面板，不适合我想要的个人站。我想把它重建成浅色纸面、学习笔记型首页。

这个目标先在对话里被澄清，再发布到 Linear，然后由 worker 在 VPS 上执行。过程中任务被拆成几步：

- 让源码分支适合 worker PR；
- 实现新的浅色学术笔记风格首页；
- 修复发布管线，保证 GitHub Pages 需要的是静态产物；
- 把生成后的静态 bundle 发布到 `main`，并验证自定义域名不再 404。

这次成功让我第一次看到完整闭环的价值。不是单次生成代码，而是从产品想法、任务拆解、后台执行、PR 审查、线上验证到复盘，形成了一条可以重复使用的路径。

网站恢复正常以后，我意识到：这套系统确实可以把自然语言想法推进到真实网站上的变化。

## 审计记录：每次运行都应该留下证据

最近我把 Native Symphony 的复盘机制又往前推了一步。之前 worker 已经会写 progress、run summary、acceptance review 和 Linear 评论；现在 run summary 升级成了更明确的审计记录。

新的 run summary 保留兼容路径：

```text
logs/runs/<issue>.json
```

同时每次运行也会写入不可变历史记录：

```text
logs/runs/history/<issue>/<run_id>.json
```

这样做的原因很简单：`logs/runs/<issue>.json` 适合当 latest record，被现有工具读取；但如果它每次覆盖，就不适合作为长期审计证据。历史目录可以保留每一次运行的上下文，不会因为下一次重试而丢失。

新的记录里会包含：

- `schema_version: 2`；
- `run_id`、`attempt`、`trigger`；
- 任务、workspace、开始和结束时间；
- outcome、defect categories、validation 摘要；
- Codex 命令、return code、stdout/stderr 尾部和最终总结；
- progress log path 和 acceptance review path；
- repair attempts；
- 运行前后的 Git snapshot；
- 最终 Linear action；
- latest 和 history 两个 record path。

这件事借鉴了 ClawSweeper 的一个核心思想：评论和页面只是展示面，真正可靠的是 durable record。只有把每次运行写成可复盘的记录，后续的 review、repair 和规则改进才有证据来源。

## 可执行建议：复盘不只是总结

daily review 以前已经会统计缺陷类型，例如：

- `task_unclear`：任务本身不够清楚；
- `env_or_permission`：环境、权限、登录、API key 等问题；
- `implementation_failed`：实现失败；
- `validation_missing`：验证不足；
- `scope_drift`：范围漂移；
- `git_or_publish`：Git、PR 或发布问题。

这些分类不是为了给 agent 打分，而是为了知道系统应该怎么进化。

现在 daily review 又加了一层：`Actionable Follow-ups`。它不只说“今天有 validation_missing”，而是直接给出建议动作类型：

- `rerun`：可以重跑；
- `create_repair_task`：应该创建一个聚焦的修复任务；
- `update_worker_rule`：应该修改 worker 规则；
- `update_reviewer_prompt`：应该修改验收 reviewer prompt；
- `human_review`：需要人工判断，不应该自动继续。

这让复盘更接近一个小型控制面。失败任务不会只是停在“失败了”，而是被路由到下一步动作。重复出现的缺陷也可以进入 memory candidates，等待人工审核后再沉淀到 skill、worker rules 或 reviewer prompt。

我希望这个机制保持克制。raw logs 是证据，不是记忆；daily review 是复盘，不是规则；长期记忆候选必须有重复证据或高影响事故支撑。

## 受控 repair loop：增强单次工作能力

另一个最新变化是受控 repair loop。

之前的流程已经是：

```text
implementation Codex
-> acceptance reviewer Codex
-> optional one repair pass
-> second acceptance review
-> commit / push / draft PR only after reviewer pass
```

现在这个 repair pass 被加上了更明确的策略门禁。

只有当 acceptance reviewer 给出 `needs_fix`，并且任务是低风险或普通风险，同时没有权限、secret、publish、dirty workspace、git blocker 或任务范围不清的问题时，worker 才会自动做一次小修。

如果问题属于下面这些类型，就不会自动 repair：

- 缺少登录、权限、token、API key；
- 涉及发布、生产环境、自定义域名、`main` 分支；
- Git 状态不干净或提交、推送失败；
- 任务目的或范围不清；
- reviewer 给出的是 `fail` 而不是 `needs_fix`。

这样做的目的不是让 agent 更激进，而是让它在安全边界内提高单次工作的完成率。小问题可以由 repair loop 修掉；高风险问题则进入 Linear review 或 follow-up task。

每次 repair attempt 也会写入 run summary，包括触发原因、policy 是否允许、reviewer findings、repair 结果、第二次验收结论和剩余 blocker。这样以后复盘时可以看见：到底是第一次实现不够好，还是 repair policy 拦得太严，或者任务本身就不该自动修。

## 状态变更仍然应该保守

这套系统里，真正危险的不是让 Codex 多写几行代码，而是让自动化直接改变外部状态。

所以当前的原则仍然是：

- 只有 reviewer `pass` 后，才允许 commit、push、创建 draft PR 和更新 Linear 到 `In Review`；
- `needs_fix` 最多只触发一次受控 repair；
- `fail` 不创建 PR；
- final success 前重新检查 Git 状态；
- publish 和 merge 默认需要人工最终确认；
- Linear 仍然是用户可见的 source of truth。

这个边界很重要。它让系统可以逐渐增强，但不至于因为一次错误判断就把错误发布到线上或污染主分支。

## 学习记录进入 Obsidian vault

工作流逐渐跑通以后，我又遇到另一个问题：项目推进会产生很多过程性记录、学习笔记和阶段性理解。如果这些内容只留在对话里，它们很快会散掉；如果直接放到网站，又会过早公开，混入不成熟的想法。

所以我把 `my-cognitive-vault` 重新整理成一个 Obsidian 兼容的学习笔记仓库。它负责承接“学习和记录”：

- 项目推进中产生的事实记录；
- 学习过程中出现的概念、问题和理解；
- Codex 帮我整理的事实层、时间线和结构；
- 我自己后续补充的“我的理解”；
- 未来可能整理成公开文章的候选稿。

现在新的内容流是：

```text
真实学习 / 项目过程
-> my-cognitive-vault
-> 整理成 Obsidian 笔记
-> 判断是否 public candidate
-> 候选内容进入 HomePage
-> 网站只展示公开版本
```

这让学习和发布分开。vault 可以不完美，可以有草稿、困惑和失败尝试；网站只展示经过选择的内容。

## `/goal` 给我的启发

最近我还看到 Codex CLI 的 `/goal` 功能。它的方向很符合我的直觉：让 agent 记住一个目标，并持续运行直到目标完成或达到预算限制。

现在它已经不只是一个概念。我在 `SelfImprovingAgent` 里给它加了一层受控运行壳，真实入口是：

```powershell
python -m harness goal-lab --goal-root <path> run <sandbox-name>
```

每个 sandbox 必须先有 `GOAL.md`。`goal-lab run` 会在 sandbox 里创建 `.goal-runtime/status.json`、`run.log`、`PROGRESS.md` 和最后消息文件，然后启动 goal-worker。当前默认命令不是生产 worker，而是：

```text
codex exec --enable goals --sandbox workspace-write --skip-git-repo-check --json --output-last-message ... -C <sandbox> -
```

也就是说，`GOAL.md` 会作为输入喂给 Codex，写权限只给当前 sandbox workspace。`--wait` 适合短 smoke test；长任务可以后台运行，再用 `runtime-status` 看 `running / completed / failed / stalled / stopped`、PID、最后日志行和停滞状态。

我也把复杂任务里的 subagent 使用纳入观察范围。`goal-lab trace` 会从 Codex JSONL 日志里提取 `spawn_agent` / `wait` 事件，让我看到 goal-worker 是否派生了 subagent、派生了几个、每个子任务的大致 prompt。这里的原则是不自研 subworker 调度器，而是让 Codex `/goal` 自己决定是否需要内置 subagent；外层系统只负责 sandbox 生命周期、运行证据、停滞识别和人工审核入口。

本机配置也有一个明确边界：如果没有设置 `GOAL_LAB_ROOT`，默认 goal-root 是 `SelfImprovingAgent` 仓库下的 `.harness/goal-lab`；在 VPS 上才建议显式使用 `/home/symphony/goal-lab`。`symphony-control` 里还有一个独立的 `.goal-lab/config.toml`，里面只启用 `[features] goals = true`，用于隔离测试，不保存生产 worker 状态。

所以当前做法是把 `/goal` 放进 goal lab，而不是直接并入正式 worker。正式工作流仍然依赖 Linear、worker、reviewer 和 PR；`/goal` 的产物默认禁止导出，必须先进入 `Goal Idea Incubator` 审核 issue。人工通过以后，才允许复制到 GitHub、vault、HomePage、正式 workspace 或普通 Linear 任务。

这也符合我对这套系统的原则：新能力可以引入，但不能让正在工作的主流程失去可控性。

## 从终端面板到消息式监控

在处理长时间 `/goal` 或 `goal-loop` 任务时，我最初尝试过 SSH 进 VPS，打开 `tmux` 面板观看进度。这个方式能看到很多细节，例如当前 cycle、worker 日志、reviewer 结论和最近产物，但它很快暴露出两个问题：

- 面板为了刷新会不断闪动，不适合长时间阅读；
- 信息量过大，容易变成另一种形式的日志堆砌。

因此监控层又被拆出来：长任务本身继续在 VPS 的 sandbox 中运行，但观察方式从“人盯终端”改成“独立监控 Codex 读状态，再用短消息汇报”。

为此 `SelfImprovingAgent` 增加了一个精简状态出口：

```bash
python3 -m harness goal-loop --goal-root /home/symphony/goal-lab brief <goal-id>
```

`brief` 不输出完整日志，而是只给监控 agent 需要的结构化信息：

- 当前状态和 active cycle；
- 推断阶段，例如 worker 正在运行、worker 已结束但 reviewer 还没写 verdict；
- 当前 bounded task；
- 最近更新的文件；
- 最新 reviewer verdict、findings 和 blockers；
- 最近几轮 cycle timeline；
- 下一次建议检查的条件。

这样，监控 agent 可以每隔一段时间读取 `brief`，再把结果压缩成一条中文消息，例如：

```text
当前 Cycle 3 worker 已写出日志，reviewer 还没给出第 3 轮 verdict。
sandbox 文件数增加，说明这一轮有实际产出。下一次重点等 cycles/003-reviewer.json 出现。
```

这次变化改变的是人机界面，而不是执行权限。`tmux` 面板仍然可以作为调试 fallback；真正默认的观察方式变成消息式监控。长任务不用再要求我一直守在一个闪动的窗口前，而是由一个只读监控 agent 把“有无进展、当前卡点、下次看什么”主动发回来。

## 我希望最终怎样使用它

理想状态下，我每天的使用方式会很简单：

1. 我用自然语言说出一个想法；
2. Codex 帮我把它澄清成 Goal Spec 和 Linear 任务；
3. VPS worker 后台领取任务；
4. Codex CLI 在对应 repo 中完成实现；
5. 如果任务很长，监控 Codex 定期读取 `brief`，用短消息告诉我关键进展；
6. reviewer agent 根据目的和证据验收；
7. GitHub PR 出现，Linear 收到结果；
8. 我验收产品，决定合并、修改或继续拆任务；
9. 系统把这次过程写入审计记录和 daily review；
10. 如果出现可修复小问题，受控 repair loop 尝试一次；
11. 如果出现重复缺陷，进入可审核的长期规则候选；
12. 如果过程产生值得保留的学习内容，先进入 `my-cognitive-vault`；
13. 如果某篇笔记适合公开，再整理成 HomePage 文章。

这样一来，项目推进和学习记录就不再是两套互相割裂的东西。项目会产生笔记，笔记会沉淀理解，成熟内容再进入网站。

## 我的理解

一套好的工作流应该高效、可靠、可进化。依靠 Codex 强大的任务执行能力，只需要用清晰可执行的指令描述需求，就能在较短时间内拿到成果。所以，目前最需要提升的部分就是如何定义和达到清晰，这是这套工作流需要长期努力的方向。

另一个同样值得努力的方向，是降低实现过程消耗的精力，把中间过程尽量交付给 Codex，把人的思考留在行动前和验收后。项目和学习都需要留下痕迹，否则很容易陷入空想。多回顾和反思，也能促成进步。

## 我还没想明白的地方

如何进一步把自己从实现中剥离出来，同时保证任务真正高质量地交出去？

现在的答案还只是阶段性的：我需要继续提高任务入口的清晰度，增强审计记录和验收标准，让系统在失败后不是简单重试，而是知道该修、该停、还是该让我重新判断。
