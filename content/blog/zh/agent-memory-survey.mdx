> 本文是对 Pengfei Du, *Memory for Autonomous LLM Agents: Mechanisms, Evaluation, and Emerging Frontiers* 的中文学习可视化笔记。原文见 [arXiv:2603.07670](https://arxiv.org/abs/2603.07670)，原文页面标注许可为 CC BY 4.0。本文为非官方改写、总结与可视化整理，部分插图由 OpenAI 图像生成工具生成，不代表原作者观点。

我把这篇论文另做了一份 [HTML 视觉导读](/visual-notes/agent-memory-survey/index.html)，适合先看图建立直觉；如果想按站点内的图像笔记阅读，也可以看 [图像笔记页](/zh/visual-notes/agent-memory-survey/)。
这篇综述真正关心的，不是“怎么给大模型外挂一个向量库”，而是一个更系统的问题：当 LLM 不再只是回答一次问题，而是要持续作为 agent 工作时，它需要什么样的记忆结构，才能不反复从零开始，不重复犯错，也不被陈旧、错误或无关的信息拖垮。作者把 agent memory 放在一个很清晰的位置上：记忆不是模型之外的装饰，而是让模型从临时文本生成器变成长期自适应系统的关键部件。

论文开头先把问题感讲得很具体。一个普通聊天机器人只要处理当前对话就够了，但一个代码助手、个人日程助手、游戏 agent 或科研 agent 都不一样。代码助手应该知道这个项目里哪些 API 不稳定，哪些修复方案已经失败过；个人助手应该记得用户稳定的偏好，而不是每次都重新问；游戏 agent 应该积累技能和配方，而不是每一局都重新摸索。作者借这些例子说明，长期 agent 的核心困难不是“模型会不会说话”，而是它能不能把过去发生过的事情转化成以后行动时可用的内部状态。

这篇文章最重要的抽象是 `write-manage-read`。很多人想到记忆，会自然想到“把历史存下来”，但作者强调，写入只是第一步。一个真正有用的记忆系统必须决定什么值得存，存进去之后如何去重、压缩、打标签、处理矛盾、更新版本，甚至在合适的时候删除。读取也不是简单地按相似度拿出几条记录，而是要在当前任务、目标、时间和风险约束下，把真正有帮助的信息送回 agent 的工作上下文。也就是说，记忆不是一个被动档案库，而是和 agent 的感知、决策、行动持续耦合的循环。

为了把这个领域梳理清楚，作者用了三个维度做分类。第一个维度是时间范围，类似人类认知里的工作记忆、情景记忆、语义记忆和程序性记忆。工作记忆就是当前上下文窗口里能直接看到的内容；情景记忆是具体经历，比如某次工具调用、某轮对话、某个环境观察；语义记忆是从很多具体经历中抽象出来的稳定知识，比如“用户偏好某种日期格式”；程序性记忆则是可复用的技能、流程或代码模式。这个分类很有帮助，因为它提醒我们：长期 agent 需要的不只是一堆对话记录，而是不同层次的记忆协同工作。

第二个维度是记忆的存储形式。最简单的是把信息直接放进 prompt，比如系统提示、最近对话、scratchpad、滚动摘要。这种方式透明、便宜、容易调试，但容量有限，而且会被上下文窗口和注意力分配限制。更常见的是向量库，把每条记录嵌入成向量，通过相似度检索出来。它能扩展到大量记录，却不擅长表达结构关系。还有一类是 SQL、key-value、知识图谱这类结构化存储，它们更适合表达时间、因果、实体关系和权限边界，但需要更明确的 schema。对于实际系统来说，作者认为混合结构往往是常态：当前上下文、向量库、结构化数据库、冷存储和技能库可能同时存在。

第三个维度是控制策略，也就是谁决定记忆怎么写、怎么读、怎么删。最朴素的是启发式规则，比如每 N 轮总结一次、检索 top-k、超过一定时间就过期。这类方法稳定可控，但不懂语境。更灵活的是把记忆操作暴露成工具，让 LLM 自己决定什么时候调用，例如搜索历史、追加核心记忆、归档文档。最新的方向则是把记忆操作当成可以学习的 policy action，通过监督学习或强化学习让 agent 学会何时存、何时取、何时更新、何时总结、何时丢弃。作者对这个方向很重视，但也承认它成本高、解释性弱，尤其是“学会遗忘”会带来安全和可靠性问题。

在具体机制上，文章先讨论上下文内记忆和压缩。把历史塞进 prompt 看起来直接，但当历史变长，系统只能滑动窗口、滚动总结或层级总结。这里最大的问题是 summary drift：每次总结都会丢掉一些低频细节，多次压缩后，agent 记住的可能是一段看似合理、但关键边界条件已经丢失的历史。作者举的典型风险是，某条很少出现但很重要的限制可能在几轮总结中消失，例如“不要直接调用生产数据库”。更大的上下文窗口只能推迟问题，不能解决问题，因为长上下文还会带来注意力稀释和成本增加。

接着文章讨论检索增强型记忆。RAG 在普通知识问答里检索的是外部文档，但在 agent 场景里，记忆库里存的是活的交互记录：工具调用日志、用户纠正、环境观察、部分计划、失败复盘。这里的难点变成三个。第一是粒度：存得太细会碎片化，存得太粗会噪声太多，所以多粒度检索变得重要。第二是查询构造：用户当前说的句子往往不是一个好 query，例如“为什么又崩了”真正需要找到的是两天前的 crash log，而不是语义上最像的一句话。第三是相关性不等于有用性，向量相似度只能说明“看起来像”，不能保证“对当前行动有帮助”。

反思型记忆是另一条重要路线。Reflexion 的想法很简单：失败之后让 agent 写一段自然语言复盘，下次再试时把这段复盘放回上下文。Generative Agents 进一步把观察、反思、规划连接起来，让模拟角色能够维持长期一致的行为。ExpeL 这类工作则从成功和失败轨迹中提取经验规则。作者认为反思记忆很有力量，因为它让 agent 不只是存事实，还能沉淀教训；但它也很危险，因为错误反思会长期强化错误信念。如果 agent 错误地总结出“某个 API 参数总是失败”，它以后可能再也不尝试这条路径，从而失去纠正自己的机会。因此，反思需要证据链、置信度、过期机制和矛盾检查。

层级记忆和虚拟上下文管理则把问题类比到操作系统。MemGPT 的思路是，当前上下文像 RAM，历史消息数据库像磁盘，长期文档库像冷存储。agent 通过记忆管理函数在不同层之间搬运信息。这个比喻很直观，因为 LLM 的上下文窗口确实像一种稀缺的高速缓存。但真正难的是调度：什么时候把哪些记忆调入当前上下文，什么时候归档，什么时候搜索冷存储。如果调度错了，系统通常不会报错，只是回答质量悄悄变差。这种静默失败是长期 agent 很难调试的根源之一。

文章也讨论了学习型记忆管理，代表是 AgeMem 这类工作。它把 store、retrieve、update、summarize、discard 这些操作都看成 agent 可调用的动作，然后用强化学习优化整体任务表现。这个方向的意义在于，记忆管理不再只是工程师手写规则，而是可以通过任务反馈学习出来。作者提到，学习到的策略可能会出现一些非显然行为，比如在上下文快满之前主动总结，或者丢弃语义相似但没有新增信息的记录。不过这条路线仍然有明显不确定性：长程 RL 很贵，跨任务迁移不稳，策略难解释，而且一旦学会错误遗忘，可能删除安全关键事实。

在评估部分，作者的态度很明确：不能只用传统检索指标评价 agent memory。Precision@k 或 nDCG 能告诉你有没有检索到“正确文档”，但不能告诉你 agent 有没有正确使用它，也不能告诉你这次检索值不值得增加延迟。更重要的是，传统指标很少处理过期、矛盾、遗忘质量和治理合规这些问题。文章把 LoCoMo、MemBench、MemoryAgentBench、MemoryArena 放在一起比较，指出这个领域正在从静态 recall 走向多会话、任务内、和行动强绑定的评测。尤其值得注意的是 MemoryAgentBench 把选择性遗忘列为能力之一，而 MemoryArena 显示很多在被动回忆任务上表现很好的系统，一旦进入完整 agentic task，表现会明显下降。

从这些 benchmark 中，作者总结出几个对研究方向很关键的判断。第一，长上下文不是记忆。即使上下文窗口扩到很大，模型也没有天然获得跨会话持久存储、结构化组织、选择性检索和删除治理。第二，RAG 有帮助，但瓶颈已经从“有没有存储”转向“能不能取出真正有用、不过期、不冲突的内容”。第三，遗忘几乎没有被充分评估，尽管任何长期部署都会遇到旧信息污染检索的问题。第四，参数化记忆和非参数化记忆各有失败方式：写进模型权重的知识使用起来自然，但难以删除和审计；外部记忆可检查、可治理，却可能被模型忽视或使用不稳定。

应用场景部分很能帮助你判断“哪里真的需要 memory”。一次性翻译工具几乎不需要长期记忆，但长期项目协作 agent 没有记忆就很难工作。个人助手最依赖语义记忆，因为它要记住用户偏好和稳定信息，但这里的核心张力是个性化和越界之间的平衡。代码 agent 需要记住架构决策、bug 历史、代码风格和验证过的解决方案，难点是代码库规模巨大，记忆不只是对话记录，还要覆盖文件、模块和工程上下文。游戏 agent 强调技能复用和长期规划。科研 agent 则需要把假设、实验结果、文献证据和不确定性一起管理。多 agent 协作又引入共享记忆、私有记忆、并发写入和权限边界这些问题。

工程部分是这篇综述最实用的地方。作者反复提醒，生产级记忆系统不能把所有交互原样存下来。写入路径需要过滤低信号内容，规范化日期、人名和数量，去重，给记录打优先级，并附上时间、来源、任务标签和置信度。读取路径也不应该每一步都跑完整检索，可以用两阶段检索、是否检索的 gate、token 预算分配和高频偏好缓存。长期系统还必须处理 staleness、contradiction 和 drift：用户地址变了，旧地址不能继续污染未来决策；用户偏好更新了，旧偏好不能和新偏好并列地被检索出来。

这里也能看出作者为什么把治理放进 memory 设计目标。记忆系统会保存隐私、偏好、工作内容甚至健康和财务信息，因此必须支持权限隔离、保留期限、删除请求和审计日志。如果记忆只是外部数据库，至少还有机会定位和删除；如果记忆已经进入模型权重，问题就复杂得多，需要 machine unlearning，而这仍然不成熟。作者推荐真实系统从“当前上下文 + 外部检索库”这种相对可控的架构开始，先做好日志和可观测性，再考虑更复杂的层级记忆或学习型控制。

最后的开放问题部分，和你前面关心的“遗忘”高度相关。作者认为当前系统在两个极端之间摇摆：要么什么都存，最后被噪声淹没；要么压缩太激进，丢掉少见但关键的事实。更好的方向可能是类似生物记忆巩固：新记忆先进入热缓冲区，经过验证、去重、重要性评估后，再晋升为长期记忆。检索也需要从语义相似走向因果相关，因为很多任务里真正有用的记忆并不一定和当前问题表面相似。反思需要可信化，否则会把错误变成长期偏见。遗忘则不应该被看成 bug，而应被看成效率、隐私和鲁棒性的必要能力。

如果把这篇文章压缩成一个研究判断，我会说它的核心不是提出一个新算法，而是把 agent memory 从“外挂工具”重新定义为一个系统工程问题。它要求我们同时考虑写入、读取、更新、压缩、检索、遗忘、权限、评估、成本和可观测性。对于想做 agent 的人，这篇文章的价值在于它能防止一个常见误区：以为接一个向量库就等于有了长期记忆。作者真正想说的是，长期可靠的 agent 需要的是一套记忆生命周期管理系统，而不仅是存储系统。

对于你现在关注的“agent 记忆是否不擅长遗忘”，这篇文章给出的答案很清楚：是的，而且这不是边缘问题。当前系统通常更擅长存储、检索、总结和反思，不擅长判断哪些信息已经过期、哪些推论应该撤销、哪些用户信息应该被遗忘、哪些派生摘要和索引也必须一起清理。选择性遗忘被作者放在开放挑战中，说明它还没有成熟范式。如果你要继续顺着这个方向读，我会把 MemoryAgentBench、FadeMem、FSFM、Zep/Graphiti 和 AgeMem 放在下一批，因为它们分别从评测、遗忘机制、时间知识图谱和学习型策略几个角度接近这个问题。

这篇综述最值得保留的一句话式理解是：未来 agent 的可靠性，不只取决于底座模型有多强，也取决于它如何管理自己曾经知道、现在仍该相信、以及应当忘掉的东西。模型选择当然重要，但对于长期 agent 来说，记忆架构可能同样重要，甚至在某些任务里更重要。
